## The core divergence in one sentence

`Paginator` is `MutableList`-like — pages are addressed by an `Int` index and distances are
well-defined. `CursorPaginator` is `LinkedList`-like — pages are addressed by a
`CursorBookmark(prev, self, next)` triple and you can only move to a cached page's immediate
neighbours. Every divergence below is a consequence of that one design choice.

---

## Behavioral differences (with reasoning)

This section lists the places where **the same-named (or analogous) method behaves differently
at runtime**. Ordered roughly by blast radius.

### 1. `restart()` semantics

**Offset `Paginator.restart()`**

- Always jumps to `BookmarkInt(page = 1)`.
- Preserves the old page-1 state before clearing the cache: if `getStateOf(1) != null` before
  `cache.clear()`, the same state is re-inserted after the clear. This is mainly a tiny
  optimisation: when the loader re-returns the same data, the snapshot machinery has something
  to diff against during the progress → result transition.
- Does not touch `persistentCache` (L2 survives restart).

```kotlin
val paginator = MutablePaginator<Item> { page -> LoadResult(api.getPage(page)) }
paginator.restart()   // always = jump(BookmarkInt(1))
```

**Cursor `CursorPaginator.restart()`**

- If `initialCursor != null`, the anchor cursor is used as the target. The paginator loads
  that specific cursor and wires `prev`/`next` from the server response.
- If `initialCursor == null`, `load.invoke(this, cursor = null)` is called — the server is
  expected to return the first page of the feed.
- **Cache is always fully cleared first.** Unlike the offset variant, nothing is preserved —
  cursor keys are server-provided, so there's no stable "page 1" to keep around.
- A **synthetic sentinel cursor** (`__cursor_paginator_progress_sentinel__`) is temporarily
  inserted to emit a `ProgressPage` before the network call. It's removed immediately when
  the real bookmark arrives (or when the load fails).

```kotlin
val paginator = mutableCursorPaginator<Message> { cursor -> /* … */ }

// A) Without anchor — server returns the feed head
paginator.restart()

// B) With anchor — jump to a known cursor instead of "page 1"
paginator.initialCursor = CursorBookmark(prev = null, self = "msg_42", next = null)
paginator.restart()
```

**Why the difference?** The offset paginator has a universally-valid "start here" address
(`page = 1`). The cursor paginator doesn't — the head of the feed is server-chosen and its
opaque key isn't known until the first load. The `initialCursor` field is the cursor-world
equivalent of "I know where to start"; when it's absent, the server decides.

### 2. `goNextPage()` implicit bootstrap drops the `loadGuard`

**Offset**

```kotlin
suspend fun goNextPage(
    …,
    loadGuard: (page: Int, state: PageState<T>?) -> Boolean = { _, _ -> true },
): PageState<T>
```

If the paginator isn't started, it falls back to `jump(BookmarkInt(1), …)` and **passes the
same `loadGuard`** through. The guard is called with `(1, currentState)`.

**Cursor**

```kotlin
suspend fun goNextPage(
    …,
    loadGuard: (cursor: CursorBookmark, state: PageState<T>?) -> Boolean = { _, _ -> true },
): PageState<T>
```

If the paginator isn't started, `goNextPage()` falls back to `restartInternal(…, loadGuard =
{ _, _ -> true })` — **your guard is bypassed**.

**Why the difference?** The cursor `loadGuard` signature takes a **non-null** `CursorBookmark`.
During implicit bootstrap there is, by definition, no bookmark yet (the server is about to
hand you the first one). Plumbing a `(CursorBookmark?, …)` guard through the whole navigation
API would pollute the typical-case signature. The explicit restart path has its own nullable
guard (`(CursorBookmark?, PageState<T>?) -> Boolean`) for callers who need it:

```kotlin
paginator.restart(loadGuard = { cursor, state ->
    cursor == null || isAllowedToBootstrap()    // the null case = initial bootstrap
})
```

**Practical implication.** If you rely on `loadGuard` to gate the very first load (e.g. wait
for auth), call `restart()` explicitly rather than relying on `goNextPage()` to bootstrap.

### 3. End-of-feed signalling

**Offset** — `FinalPageExceededException`:

```kotlin
class FinalPageExceededException(
    val attemptedPage: Int,
    val finalPage: Int,
)

paginator.finalPage = 100
try {
    paginator.goNextPage()
} catch (e: FinalPageExceededException) {
    // attemptedPage=101, finalPage=100
}
```

The cap is an **explicit upper bound** set by the app (typically from backend metadata). It
triggers on `attemptedPage > finalPage`.

**Cursor** — `EndOfCursorFeedException`:

```kotlin
class EndOfCursorFeedException(
    val attemptedCursorKey: Any?,
    val direction: Direction,   // FORWARD | BACKWARD
)

try {
    paginator.goNextPage()
} catch (e: EndOfCursorFeedException) {
    // attemptedCursorKey = "p4" (the current tail), direction = FORWARD
}
```

The boundary is **implicit** — detected when the pivot cursor has `next == null` (or
`prev == null` for `goPreviousPage`). No configurable cap; the server decides.

**Why the difference?** In an offset world "how many pages are there" is meaningful and
often known up-front. In a cursor world the feed often has no known length — the server
streams you a `nextCursor` until it stops. That "no more cursor" signal is the only honest
end marker, and it's directional: you can reach the head and the tail independently, so the
exception carries a `direction`.

**Practical implication.** On error-handling code-paths, catch **both**:

- in offset UI: `FinalPageExceededException` → "reached last page" hint.
- in cursor UI: `EndOfCursorFeedException` with `direction == FORWARD` → "you're caught up".

### 4. `removeState`: neighbour re-linking vs page collapse

**Offset `MutablePaginator.removeState(pageToRemove)`**

The offset implementation has to maintain the `page: Int` invariant — when you remove page 5
from a chain of pages `1,2,3,4,5,6,7`, the pages that were 6 and 7 must become 5 and 6.
Otherwise, the `page` numbers in the cache stop mapping 1:1 to server pages.

The algorithm:

1. Find the state of `pageToRemove` in the cache.
2. **Collapse** consecutive pages: for every page `> pageToRemove`, rewrite
   `state.page = state.page - 1`.
3. Detect gaps in the page sequence (e.g. `1,2,3, [gap], 11,12,13`) and handle each contiguous
   chunk separately, because collapsing must not close a gap.
4. Recalculate the context window: if `pageToRemove` was inside the window, shift it via
   `findNearContextPage`.

```kotlin
val paginator = mutablePaginator<String> { /* … */ }
// cache = [page=1, page=2, page=3]
paginator.removeState(2)
// cache = [page=1, page=2] where the second entry is what was page 3
//                          (its `page` field was rewritten from 3 → 2).
```

**Cursor `MutableCursorPaginator.removeState(selfToRemove)`**

The cursor implementation doesn't have to rewrite anything. Each page carries its opaque
`self` key, and the chain order is defined by the `prev`/`next` links on the surviving
neighbours.

The algorithm:

1. Find the removed bookmark and its two neighbours.
2. Drop the entry from the cache.
3. Rewrite the neighbours' links: `prev.next = removed.next`, `next.prev = removed.prev`.
4. If the removed cursor was `startContextCursor` / `endContextCursor`, shift the boundary to
   the appropriate neighbour. If the neighbour doesn't exist in the cache, the boundary
   becomes `null`.

```kotlin
val paginator = mutableCursorPaginator<String> { /* … */ }
// chain: A → B → C → D
paginator.removeState("B")
// chain: A → C → D
// A.next = "C"  (was "B")
// C.prev = "A"  (was "B")
// A's and C's own `self` keys are untouched.
```

**Why the difference?** The offset version's `page` field IS the identity AND the order. You
can't change one without the other. The cursor version keeps identity (`self`) and order
(`prev`/`next`) in separate fields, so a removal only affects the order axis.

**Practical implication.** `removeState` is **O(1)** in the cursor variant but can be **O(N)**
in the offset variant (where N is the number of pages after the removed one). For feeds with
many pages, prefer the cursor paginator when removals are common.

### 5. `addAllElements` overflow cascade

Both variants cascade overflow elements along the chain when an insert pushes a page past
`capacity`. They differ in **what happens at the tail of the cached chain**.

**Offset `MutablePaginator.addAllElements`**

When the cascade runs out of same-class pages to push into (e.g. hits a gap or the end), the
offset version does **not** drop the overflow. Instead it preserves a global invariant
("shift by N") by **peeling N items off the head of the next chunk of same-class pages** and
feeding them back into the cascade:

```
Before: 1..5 [gap] 10..15 [gap] 20..25   (each page capacity=5)
Insert: 5 new items at page 13

After:  1..5                          — untouched
        10..12                        — untouched
        13..15 shifted by +5          — standard cascade
        page 20 loses its last 5 items (partial)
        21..25 shifted by +5          — still full
        page 26 newly created with page 25's original tail
```

No bookmark factory is needed, because the offset page number (`page + 1`) is always
meaningful — synthesising a new page at `targetPage + 1` just copies the same `page: Int + 1`
into the new state.

**Cursor `MutableCursorPaginator.addAllElements`**

The cursor version has no numeric `page + 1` to synthesise, so there are **two opt-in modes**:

```kotlin
fun addAllElements(
    elements: List<T>,
    targetSelf: Any,
    index: Int,
    bookmarkFactory: CursorBookmarkFactory? = null,
    initPageState: ((previous: CursorBookmark, data: List<T>) -> PageState<T>)? = null,
)
```

1. **Drop (default, `bookmarkFactory = null`).** When the cascade hits a missing `next` link,
   the remaining overflow elements are silently dropped (a debug log line is emitted). Typical
   for feeds where an overflowed page will be re-fetched from the server anyway on the next
   scroll.
2. **Spawn a new tail (opt-in, `bookmarkFactory != null`).** The factory mints a fresh
   `CursorBookmark` (its `self` key is your responsibility) and a new tail page is inserted
   into the cache, linked to its predecessor:

```kotlin
paginator.addAllElements(
    elements = listOf(msg1, msg2, msg3),
    targetSelf = "msg_100",
    index = 0,
    bookmarkFactory = { overflowIndex, previous ->
        // overflowIndex counts cascade steps; previous is the new page's parent.
        CursorBookmark(
            prev = previous.self,
            self = "overflow-${previous.self}-$overflowIndex",
            next = null,
        )
    },
)
```

Note the cursor variant does **not** have the offset variant's "peel from next chunk" fallback
— because there's no guaranteed chain across gaps. The cascade either fits or it doesn't.

**Why the difference?** The offset variant's fallback exists to preserve the "page number =
index in data" invariant across gaps. The cursor variant doesn't have that invariant (cursors
don't carry order across gaps), so the fallback wouldn't have anything to preserve. Instead
it offers a clean opt-in to extend the chain.

**Practical implication.** If you rely on overflow being preserved in the cursor world, pass
a `bookmarkFactory`. Otherwise your inserts are capacity-bounded.

### 6. Bookmark visibility filter

`jumpForward` / `jumpBack` skip bookmarks that are "already visible on screen" (to avoid
no-op jumps). The visibility check differs:

**Offset** — range containment:

```kotlin
val visibleRange: IntRange? = core.snapshotPageRange()
if (visibleRange != null && candidate.page in visibleRange) {
    // skip — already visible
}
```

**Cursor** — exact-match on `self` keys visible in the snapshot:

```kotlin
val visibleSelves: Set<Any> = core.snapshotSelves().toHashSet()
if (candidate.self in visibleSelves) {
    // skip — already visible
}
```

**Why the difference?** `IntRange` is trivial for numeric pages but meaningless for opaque
cursors. The cursor variant instead materialises the set of `self` keys that sit on the chain
between `startCursor` and `endCursor` of the last emitted snapshot.

**Practical implication.** For large context windows the cursor check is **O(window size)**
per candidate vs O(1) for the offset `in IntRange`. In practice `snapshotSelves()` is called
once per `jumpForward` / `jumpBack` and hashed into a set, so this is still amortised well.

### 7. `syncBookmarkIndex`: ordered vs exact match

After a successful jump / restart, both paginators sync the bookmark iterator so that the
next `jumpForward` starts from the right position.

**Offset**

```kotlin
private fun syncBookmarkIndex(page: Int) {
    val index = bookmarks.indexOfFirst { it.page > page }
    bookmarkIndex = if (index == -1) bookmarks.size else index
}
```

Relies on **numeric ordering** of bookmark pages. Places the index right after the last
bookmark whose page is `<=` the current page.

**Cursor**

```kotlin
private fun syncBookmarkIndex(cursor: CursorBookmark) {
    val index = bookmarks.indexOfFirst { it.self == cursor.self }
    bookmarkIndex = if (index == -1) bookmarks.size else (index + 1)
}
```

Uses **exact-match** — puts the iterator right after the bookmark whose `self` was reached.
If the current cursor isn't one of the saved bookmarks, the iterator parks at the end.

**Why the difference?** Cursors have no natural ordering. "The bookmark after this page"
requires either a known order (which we don't have) or an exact match (which we do).

**Practical implication.** Keep bookmarks in the **order you want to traverse** in the cursor
paginator. The offset paginator sorts by page number implicitly; the cursor one respects your
insertion order.

### 8. Default bookmarks list

- **Offset:** `bookmarks = MutableList(initial = [BookmarkInt(1)])`.
- **Cursor:** `bookmarks = MutableList(initial = [])`.

**Why the difference?** `BookmarkInt(1)` is universally valid for offset paginators (every
offset-paged API exposes page 1). There is no equivalent for cursor — the head cursor key is
opaque and server-provided, so the paginator can't pre-seed anything sensible.

**Practical implication.** `jumpForward()` on a freshly-built cursor paginator returns `null`
immediately. Pre-populate `bookmarks` if you want it to do anything.

### 9. `release()` defaults

**Offset `Paginator.release()`**

```kotlin
fun release(capacity: Int = DEFAULT_CAPACITY, silently: Boolean = false) {
    core.release(capacity, silently)
    bookmarks.clear()
    bookmarks.add(BookmarkInt(page = 1))     // re-seed
    bookmarkIndex = 0
    finalPage = Int.MAX_VALUE                // reset cap
    lockJump = false
    /* … reset all lock flags */
}
```

**Cursor `CursorPaginator.release()`**

```kotlin
fun release(capacity: Int = DEFAULT_CAPACITY, silently: Boolean = false) {
    core.release(capacity, silently)
    bookmarks.clear()                         // no re-seed
    bookmarkIndex = 0
    initialCursor = null                      // no finalPage — release initialCursor instead
    lockJump = false
    /* … reset all lock flags */
}
```

**Why the difference?** Mirrors **default bookmarks list** and **no `finalPage` field** —
the cursor paginator has nothing to re-seed and no cap to reset. Instead it clears
`initialCursor` because that's the cursor-world anchor.

**Practical implication.** After `release()`, a cursor paginator has no bookmarks and no
anchor — calling `goNextPage()` on it boots from `load(null)`, not from a known cursor.

### 10. Snapshot auto-expansion at chain boundaries

Both variants emit snapshots that **extend one step past the context window** on each side
to include a boundary `ProgressPage` / `ErrorPage` (so the UI can render a "loading more" or
"retry" indicator adjacent to loaded content). They differ subtly in handling the chain's
absolute end:

**Offset** — walks backward through filled successes, subtracts 1, then **coerces to `>= 1`**:

```kotlin
val expandedBackwardPage: Int? =
    walkWhile(pivotBackwardState, next = { it - 1 }, predicate = ::isFilledSuccess)
        ?.page?.minus(1)
        ?.coerceAtLeast(1)
```

So if the walk reaches page 1 (the chain head), subtracting 1 gives 0, coerced to 1. No
"phantom" boundary is emitted at the head.

**Cursor** — walks backward through filled successes, then tries one more `cache.walkBackward`
(no coercion possible — there's no "minimum cursor"):

```kotlin
val backwardEdge: CursorBookmark? =
    backwardExpand?.second?.let { cache.walkBackward(it) }
val backwardBoundary: CursorBookmark = backwardEdge ?: backwardExpand?.second ?: start
```

If the walk reaches a cursor whose `prev == null`, `walkBackward` returns `null` and the
range min stays at the last filled success. Same behaviour as offset — no phantom boundary —
but achieved through a link check instead of numeric coercion.

**Why the difference?** Offset has a numeric floor (`page >= 1`). Cursor has no numeric floor
but does have a link-based one (`prev == null`). Both converge on the same observable
behaviour.

**Practical implication.** None, really — this is implementation trivia. But if you're
writing tests that inspect the internal snapshot range, be aware that the cursor version
returns a `Pair<CursorBookmark, CursorBookmark>?` (via `snapshotCursorRange()`) rather than
an `IntRange?` (via `snapshotPageRange()`).

## Functions removed in CursorPaginator

These have no cursor-world equivalent because the underlying concept doesn't translate.

### `core.resize(capacity)`

**Offset.** Redistributes every item across the entire context window into pages of the new
capacity. Essentially: flatten to a list of items, clear the cache, re-paginate.

**Cursor.** Throws `UnsupportedOperationException`.

**Why removed?** Resizing requires re-assigning page identities — "everything after index X
becomes page 2" etc. In the offset world, that's trivial because identity is just `page: Int`.
In the cursor world, identity is the server-given `self` key; the client can't mint a new
valid cursor by recomputing. Redistributing into pages would need a **made-up** cursor per
new page — which either needs a pluggable factory (you can already do this manually by
rebuilding the chain via `MutableCursorPaginator.addAllElements` with a `bookmarkFactory`) or
silently invents keys, which would break the contract that `self` comes from the server.

**Workaround.** Build a fresh cursor paginator with the new `capacity` and replay the data,
or use `MutableCursorPaginator.addAllElements(..., bookmarkFactory)` to manually re-chunk.

---

### `core.findNearContextPage`

**Offset.** Binary-search across cached page numbers for the nearest valid page to a target
range, using `|page_a - page_b|` as the distance metric.

**Cursor.** Not available.

**Why removed?** Binary search and distance metrics both require ordered, comparable keys
with a meaningful notion of "closer". Cursors have none of those properties. Asking "which
cached cursor is closest to 'msg_abc123'?" is meaningless — cursor keys are opaque strings,
numeric IDs that aren't monotonic, or timestamps whose "distance" may or may not correlate
with feed position.

**What replaces it?** Navigation to a specific target is done via `jump(bookmark)`. The
"find the closest loaded cursor" problem simply doesn't arise in the cursor model — you
either know which cursor you want to jump to, or you don't.

---

### `paginator.finalPage`

**Offset.** `var finalPage: Int = Int.MAX_VALUE` — upper bound, throws
`FinalPageExceededException` on `goNextPage`/`jump` when exceeded.

**Cursor.** Not present.

**Why removed?** Already covered in
[Difference #3 — End-of-feed signalling](#3-end-of-feed-signalling). The equivalent signal
is `next == null` on the current edge cursor. Having both a numeric cap and a null-link
sentinel would be redundant and potentially inconsistent.

**Workaround.** If you need a **soft** client-side limit (e.g. "max 100 pages even if the
server has more"), wrap `load` and count loads:

```kotlin
var loaded = 0
val paginator = mutableCursorPaginator<Message> {
    load { cursor ->
        if (loaded++ >= 100) {
            // Pretend we've hit the tail
            val last = cursor?.self ?: "first"
            CursorLoadResult(data = emptyList(), bookmark = CursorBookmark(null, last, null))
        } else {
            /* real network call */
        }
    }
}
```

### `PageExt`: `near` / `far` / `gap`

**Offset.** Infix extensions on `PageState<*>` that compute numeric distances:

```kotlin
infix fun PageState<*>.gap(other: PageState<*>): Int      // |a.page - b.page|
infix fun PageState<*>.near(other: PageState<*>): Boolean // gap in [0, 1]
infix fun PageState<*>.far(other: PageState<*>): Boolean  // gap > 1
```

Used internally by `MutablePaginator.removeState` to detect gaps in the sorted page list.

**Cursor.** These extensions *compile* (they operate on `PageState<*>.page: Int` which exists
for every `PageState`), but the `page` field in a cursor paginator's cache is a **synthetic
monotonic counter** (emitted by `CursorPaginator.syntheticPage()`), unrelated to cursor
position. So `gap`/`near`/`far` produce meaningless numbers.

**Why removed from the model (but not the type)?** Removing the `page: Int` field from
`PageState` itself would break every `@Serializable PageState` and every downstream consumer
of the offset paginator. Keeping the field (with a synthetic value in cursor-mode) was the
lesser evil; **do not** use `near`/`far`/`gap` in cursor code.

**What replaces it?** Neighbour checks via cache links:

```kotlin
// "Is `b` the next page after `a`?"  (cursor version of `a near b`)
val areAdjacent = cache.walkForward(a)?.self == b.self
```

## Signature-only differences (same logic)

These methods behave identically — only the key type changes.

| Offset method                                                                        | Cursor method                                                                                                | Key changes                                                                                                          |
|--------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------|
| `jump(bookmark: BookmarkInt)`                                                        | `jump(bookmark: CursorBookmark)`                                                                             | `Int` page → opaque `self`                                                                                           |
| `refresh(pages: List<Int>)`                                                          | `refresh(cursors: List<CursorBookmark>)`                                                                     | `Int` → cursors                                                                                                      |
| `loadOrGetPageState(page, …)`                                                        | `loadOrGetPageState(hint: CursorBookmark, cachedCursor = hint, …)`                                           | Different return: `PageState<T>` → `Pair<CursorBookmark, PageState<T>>` (cursor may be refreshed from load response) |
| `core.getStateOf(page: Int)`                                                         | `core.getStateOf(self: Any)`                                                                                 | —                                                                                                                    |
| `core.markDirty(page: Int)`                                                          | `core.markDirty(self: Any)`                                                                                  | —                                                                                                                    |
| `core.drainDirtyPagesInRange(IntRange)`                                              | `core.drainDirtyCursorsInRange(start, end)`                                                                  | `IntRange` → cursor pair                                                                                             |
| `saveStateToJson(elementSerializer)`                                                 | `saveStateToJson(elementSerializer, keySerializer)`                                                          | Need one extra `KSerializer` for the cursor-key type                                                                 |
| `PagingCache<T>`                                                                     | `CursorPagingCache<T>`                                                                                       | Keyed by `Int` → keyed by `Any`                                                                                      |
| `LruPagingCache` / `FifoPagingCache` / `TtlPagingCache` / `SlidingWindowPagingCache` | `LruCursorPagingCache` / `FifoCursorPagingCache` / `TtlCursorPagingCache` / `SlidingWindowCursorPagingCache` | Same eviction semantics, protection windows walk cursor chain instead of `IntRange`                                  |
| `PersistentPagingCache<T>`                                                           | `CursorPersistentPagingCache<T>`                                                                             | `load(page)` → `load(self)` returning `Pair<CursorBookmark, PageState<T>>?`                                          |

## Fully identical mechanisms

These work *exactly* the same in both variants, no adaptation needed.

- **`PageState` sealed hierarchy.** `SuccessPage`, `ProgressPage`, `ErrorPage`
  and `PlaceholderPageState` are shared. Custom subclasses work in both worlds.
  Empty state is `SuccessPage` with `data.isEmpty()` — use `isEmptyState()` to detect it.
- **`PaginatorUiState<T>` classifier.** The `toUiState` / `asUiState` / `uiState` extensions
  are bookmark-agnostic — they classify the snapshot purely by `PageState` type and data
  payload. The cursor paginator simply re-imports them.
- **`Interweaver` + `WovenEntry`.** The opt-in `Flow<PaginatorUiState<T>>.interweave(weaver)`
  operator works identically on either flavour.
- **`PaginatorLogger`.** Same interface, same log levels, same components.
- **Lock flags.** `lockJump`, `lockGoNextPage`, `lockGoPreviousPage`, `lockRestart`,
  `lockRefresh` and their respective exception types are shared.
- **`transaction { }` mechanics.** Savepoint + rollback + auto-flush-on-success are identical.
  Only the savepoint contents differ (stores cursors instead of page numbers).
- **Capacity + incomplete-page detection.** `capacity`, `isCapacityUnlimited`, `coerceToCapacity`
  behave identically. Incomplete-page re-requests work the same way.
- **Dirty-page tracking + in-context refresh.** Same semantics, keyed by `self` instead of
  `Int`.
- **Prefetch controller lifecycle.** `prefetchController(…)`, `onScroll(firstVisibleIndex,
  lastVisibleIndex, totalItemCount)`, `cancel()`, `reset()`, `enabled` — all identical. Only
  the edge check differs (`next != null` vs `page < finalPage`).
- **DSL structure.** Sealed builder hierarchy, `@PaginatorDsl` marker, required `load { }`,
  merging `initializers { }`, `bookmarks(…)` vararg + list overloads.

## Migration cheat sheet

Porting offset code to cursor code? Here's the usual substitution table.

| You had                                           | You want                                                                         |
|---------------------------------------------------|----------------------------------------------------------------------------------|
| `Paginator<T>(load = { page -> … })`              | `CursorPaginator<T>(core = CursorPagingCore()) { cursor -> … }`                  |
| `paginator { load { page -> LoadResult(…) } }`    | `cursorPaginator { load { cursor -> CursorLoadResult(…, bookmark = …) } }`       |
| `BookmarkInt(42)`                                 | `CursorBookmark(prev = …, self = …, next = …)`                                   |
| `paginator.finalPage = 100`                       | remove — use `next == null` on the server response                               |
| `catch (e: FinalPageExceededException)`           | `catch (e: EndOfCursorFeedException)`                                            |
| `catch (e: LoadGuardedException)`                 | `catch (e: CursorLoadGuardedException)` (or the base `LoadGuardedBaseException`) |
| `paginator.core.getStateOf(5)`                    | `paginator.core.getStateOf("msg_5")`                                             |
| `paginator.core.markDirty(listOf(1, 2))`          | `paginator.core.markDirty(listOf("a", "b"))`                                     |
| `LruPagingCache<T>(maxSize = 50)`                 | `LruCursorPagingCache<T>(maxSize = 50)`                                          |
| `paginator.refresh(listOf(1, 2, 3))`              | `paginator.refresh(cursors = bookmarks)`                                         |
| `paginator.setElement(item, page = 3, index = 0)` | `paginator.setElement(item, self = "msg_3", index = 0)`                          |
| `paginator.saveStateToJson(Item.serializer())`    | `paginator.saveStateToJson(Item.serializer(), String.serializer())`              |

Patterns that **don't** have a direct port:

- `paginator.core.resize(capacity = 10, resize = true)` — not supported; rebuild the paginator
  with the new capacity, or re-chunk manually via `addAllElements` + `bookmarkFactory`.
- `paginator.core.findNearContextPage(10, 20)` — not applicable; navigate to a specific
  cursor via `jump(bookmark)`.
- `state1 near state2` — use `cache.walkForward(cursor1)?.self == cursor2.self` instead.

---

## Choosing between the two

**Use `Paginator` when:**

- The backend exposes numeric page offsets (`?page=5&size=20`).
- You need random-access jumps to arbitrary pages (`jump(BookmarkInt(42))`).
- The total number of pages is known upfront (e.g. paginated tables, search results).
- You benefit from `resize(capacity)` or `findNearContextPage`.

**Use `CursorPaginator` when:**

- The backend returns opaque continuation tokens (`nextCursor` / `prevCursor`).
- You work with GraphQL connections (`pageInfo { startCursor, endCursor, hasNext, hasPrev }`).
- The feed has no stable notion of "page 42" (chat, activity streams, infinite social feeds).
- Inserts and removals at arbitrary positions are common — cursor `removeState` is O(1), offset
  is O(N).

**When in doubt**, start with `Paginator`. The cursor variant is additive, purpose-built for
a narrower set of backends, and slightly more ceremony to use. If you later need to switch,
the page-state model, UI state, caches and CRUD surface all carry over — only the addressing
changes.
