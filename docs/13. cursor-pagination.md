## When to use it

Use `CursorPaginator<T>` when the source exposes pages through **opaque
continuation tokens** rather than numeric offsets:

- Chat feeds, activity streams, social timelines that move around a moving edge.
- GraphQL connections (`edges { node, cursor }` /
  `pageInfo { startCursor, endCursor, hasNext, hasPrev }`).
- REST APIs that return `nextCursor` / `prevCursor` without a numbered index (Slack,
  Instagram, Reddit, Stripe listings, ‚Ä¶).
- Any feed where an absolute "page 42" isn't meaningful or stable.

Use the classic [`Paginator<T>`](1.%20core-concepts.md) when pages have a stable
numeric index (`page=1..N`) and the backend supports random access.

## Mental model: `Paginator` is RandomAccess, `CursorPaginator` is LinkedList

The classic `Paginator` is `MutableList`-like: every page is addressed by a
numeric index, you can jump to any page, and distances between pages are
well-defined integers.

`CursorPaginator` is `LinkedList`-like: each page knows only its immediate
neighbours through a `CursorBookmark(prev, self, next)`. You navigate by
following links, not by computing offsets.

```
   ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê   next   ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê   next   ‚îå‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îê
   ‚îÇ  page A ‚îÇ ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∂ ‚îÇ  page B ‚îÇ ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚ñ∂ ‚îÇ  page C ‚îÇ
   ‚îÇ prev: ‚àÖ ‚îÇ ‚óÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ ‚îÇ prev: A ‚îÇ ‚óÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ ‚îÇ prev: B ‚îÇ
   ‚îÇ self: A ‚îÇ   prev   ‚îÇ self: B ‚îÇ   prev   ‚îÇ self: C ‚îÇ
   ‚îÇ next: B ‚îÇ          ‚îÇ next: C ‚îÇ          ‚îÇ next: ‚àÖ ‚îÇ
   ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò          ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò          ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
    HEAD                                         TAIL
```

`prev == null` marks the head of the feed; `next == null` marks the tail.
These are the canonical "end of feed" signals.

## CursorBookmark

```kotlin
data class CursorBookmark(
    val prev: Any?,
    val self: Any,
    val next: Any?,
) : Bookmark
```

- `self` is the identity of the page and the cache key. `equals`/`hashCode`
  look **only** at `self`, so an updated bookmark (e.g. server later returns
  a real `next` where `null` used to sit) overwrites the previous entry in
  place.
- `prev` / `next` are the `self` keys of the neighbours, or `null` at a feed
  boundary.
- `self: Any` is deliberately untyped ‚Äî use whatever your backend returns:
  opaque strings, numeric ids, ISO timestamps, composite `data class`
  tokens. Two requirements: stable `equals` and `hashCode`.

## CursorLoadResult

```kotlin
open class CursorLoadResult<T>(
    open val data: List<T>,
    open val metadata: Metadata? = null,
    open val bookmark: CursorBookmark,
)
```

Every successful load must return a `bookmark` carrying the real
`prev`/`self`/`next` links as observed by the server. `next == null` tells the
paginator "this is the tail ‚Äî don't try to paginate further".

## Quick Start

```kotlin
import com.jamal_aliev.paginator.bookmark.CursorBookmark
import com.jamal_aliev.paginator.dsl.mutableCursorPaginator
import com.jamal_aliev.paginator.load.CursorLoadResult

class ChatViewModel(private val api: ChatApi) : ViewModel() {

    private val paginator = mutableCursorPaginator<Message>(capacity = 50) {
        load { cursor ->
            val page = api.getMessages(cursor?.self as? String)
            CursorLoadResult(
                data = page.items,
                bookmark = CursorBookmark(
                    prev = page.prevCursor,      // null at the head
                    self = page.selfCursor,      // required ‚Äî cache key
                    next = page.nextCursor,      // null at the tail
                ),
            )
        }
    }

    init {
        paginator.uiState
            .onEach { uiState -> render(uiState) }
            .launchIn(viewModelScope)

        viewModelScope.launch { paginator.restart() }
    }

    fun loadMore() {
        viewModelScope.launch { paginator.goNextPage() }
    }
    fun loadEarlier() {
        viewModelScope.launch { paginator.goPreviousPage() }
    }

    override fun onCleared() {
        paginator.release()
        super.onCleared()
    }
}
```

`paginator.uiState` re-uses the exact same bookmark-agnostic mapping as the
offset paginator (see
[State ‚Üí PaginatorUiState](3.%20state.md#paginatoruistate)):
`Idle` / `Loading` / `Empty` / `Error` / `Content(items, prependState, appendState)`.

## DSL Builder

The builder block mirrors the offset
[`mutablePaginator`/`paginator`](11.%20dsl-builder.md) DSL, with a cursor
`load` signature:

```kotlin
val messages = mutableCursorPaginator<Message>(capacity = 50) {
    load { cursor -> /* ‚Ä¶ */ }

    // L1 cache ‚Äî stack eviction strategies with `+`
    cache = LruCursorPagingCache(maxSize = 100) + TtlCursorPagingCache(ttl = 5.minutes)
    persistentCache = roomBackedCursorCache

    // Optional anchor cursor for restart() ‚Äî jump to this cursor on restart instead
    // of asking the server for the "first" page.
    initialCursor = CursorBookmark(prev = null, self = "msg_123", next = null)

    recyclingBookmark = true
    logger = PrintPaginatorLogger()

    // Bookmark list for jumpForward/jumpBack (empty by default ‚Äî cursor keys are opaque)
    bookmarks(
        CursorBookmark(prev = null, self = "msg_1", next = null),
        CursorBookmark(prev = null, self = "msg_50", next = null),
    )

    initializers {
        empty { page, data, meta -> MyEmptyPage(page, data, meta) }
        error { e, page, data, meta -> MyErrorPage(e, page, data, meta) }
    }
}
```

Read-only DSL counterpart:

```kotlin
val readOnly: CursorPaginator<Message> = cursorPaginator(capacity = 50) {
    load { cursor -> /* ‚Ä¶ */ }
}
```

Everything under `builder` / `BasePaginatorBuilder` survives: `@PaginatorDsl`
marker, sealed hierarchy, required `load { }`, merging `initializers { }`.

## Navigation

### restart

Clears the cache and bootstraps the feed.

```kotlin
suspend fun restart(): Unit
```

- If `initialCursor != null`, the paginator loads **that specific cursor** and
  wires its `prev`/`next` from the server response.
- If `initialCursor == null`, `load` is invoked with `cursor = null` ‚Äî the
  server is expected to return the first page of the feed.

Typical usage: swipe-to-refresh on a chat screen.

### jump

```kotlin
suspend fun jump(bookmark: CursorBookmark): Pair<CursorBookmark, PageState<T>>
```

Navigates directly to the page identified by `bookmark.self`. The backend
response fills in real neighbour links (`prev`, `next`), so a stub of
`CursorBookmark(prev = null, self = knownId, next = null)` is valid when you
know only the target id.

Cache-hit semantics: if the target is already a filled success page, the
method returns it without touching the source ‚Äî just like the offset variant.

### goNextPage / goPreviousPage

```kotlin
suspend fun goNextPage(): PageState<T>
suspend fun goPreviousPage(): PageState<T>
```

Follow `endContextCursor.next` / `startContextCursor.prev`. When that link is
`null`, the feed boundary has been reached and the paginator throws
[
`EndOfCursorFeedException`](../paginator/src/commonMain/kotlin/com/jamal_aliev/paginator/exception/EndOfCursorFeedException.kt)
with a `direction: FORWARD | BACKWARD` marker instead of issuing another load.

`goNextPage` also **implicitly bootstraps** the feed when called before any
navigation, mirroring the offset variant which auto-jumps to page 1.

### jumpForward / jumpBack

Iterates the `bookmarks: MutableList<CursorBookmark>` list, skipping any
bookmark whose `self` is already visible in the last emitted snapshot. The
iterator index wraps around when `recyclingBookmark = true`.

### refresh

```kotlin
suspend fun refresh(cursors: List<CursorBookmark>): Unit
```

Reloads the specified cursors in parallel, preserving each page's cached data
while the `ProgressPage` is in flight. `refreshAll()` (extension) refreshes
every cached cursor.

## Mutation (MutableCursorPaginator)

### removeState

```kotlin
fun removeState(selfToRemove: Any): PageState<T>?
```

Drops the page identified by `selfToRemove` **and re-links its neighbours**:
`prev.next` is rewritten to point at `removed.next`, and `next.prev` to
`removed.prev`. No numeric-page collapse is needed ‚Äî this is where the
LinkedList model pays off.

If the removed cursor was `startContextCursor` / `endContextCursor`, the
boundary shifts to the appropriate neighbour (or becomes `null` when no
neighbor remains in the cache).

### setElement / removeElement

```kotlin
fun setElement(element: T, self: Any, index: Int, isDirty: Boolean = false)
fun removeElement(self: Any, index: Int, isDirty: Boolean = false): T
```

Work the same as the offset counterparts, keyed by `self` instead of `page`.
`removeElement` auto-refills the current page from its `next` neighbour of
the same class until either the page hits `capacity` or the successor runs
out of items ‚Äî if the successor is drained to empty, it is also removed (and
its neighbours re-linked).

### addAllElements and overflow cascade

```kotlin
fun addAllElements(
    elements: List<T>,
    targetSelf: Any,
    index: Int,
    bookmarkFactory: CursorBookmarkFactory? = null,
    initPageState: ((previous: CursorBookmark, data: List<T>) -> PageState<T>)? = null,
)
```

Inserts `elements` at position `index` on the page identified by `targetSelf`.
If the insert pushes elements past `capacity`, they **cascade along
`cursor.next`** to the next cached page of the same type.

When the cascade reaches the tail of the cached chain there are two modes:

1. **Drop (default).** Without a `bookmarkFactory`, the cascade blocks and the
   surplus is dropped. A debug log line is emitted. Suitable when "overflow
   means the data will be re-fetched anyway".
2. **Spawn a new tail (opt-in).** Pass a `CursorBookmarkFactory` that mints a
   fresh `CursorBookmark` (its `self` key is your responsibility) and a
   brand-new tail page is created in the cache, linked to its predecessor.

```kotlin
paginator.addAllElements(
    elements = listOf(msg1, msg2, msg3),
    targetSelf = "msg_100",
    index = 0,
    bookmarkFactory = { overflowIndex, previous ->
        CursorBookmark(prev = previous.self, self = "overflow-$overflowIndex", next = null)
    },
)
```

## UI state

The `uiState: Flow<PaginatorUiState<T>>` extension on `CursorPaginator` is
re-exported from the **same** bookmark-agnostic classifier that the offset
paginator uses. All semantics (`Idle` / `Loading` / `Empty` / `Error` /
`Content` with `prependState` / `appendState` boundaries) apply verbatim ‚Äî
see [State ‚Üí PaginatorUiState](3.%20state.md#paginatoruistate).

## Prefetch

`CursorPaginatorPrefetchController` behaves exactly like the offset
`PaginatorPrefetchController`
([Prefetch](7.%20prefetch.md)), except its edge check is:

- forward prefetch fires only when `endContextCursor?.next != null`;
- backward prefetch fires only when `startContextCursor?.prev != null`.

Create one via `paginator.prefetchController(scope = ‚Ä¶, prefetchDistance = ‚Ä¶)`.

## Caching

The cursor variant ships its own cache interface
`CursorPagingCache<T>` keyed by `CursorBookmark.self: Any`, plus direct ports
of every eviction strategy from the offset side:

| Offset                     | Cursor                           |
|----------------------------|----------------------------------|
| `DefaultPagingCache`       | `DefaultCursorPagingCache`       |
| `LruPagingCache`           | `LruCursorPagingCache`           |
| `FifoPagingCache`          | `FifoCursorPagingCache`          |
| `TtlPagingCache`           | `TtlCursorPagingCache`           |
| `SlidingWindowPagingCache` | `SlidingWindowCursorPagingCache` |
| `PersistentPagingCache`    | `CursorPersistentPagingCache`    |

Composition with `+` works the same way:

```kotlin
val cache = LruCursorPagingCache<Message>(maxSize = 50) +
        TtlCursorPagingCache(ttl = 5.minutes) +
        SlidingWindowCursorPagingCache(margin = 1)
```

The sliding-window variant walks `next`/`prev` links from
`startContextCursor` / `endContextCursor` (plus `margin` steps beyond each
edge) instead of comparing integer page numbers.

`CursorPersistentPagingCache` stores the full `CursorBookmark` alongside the
`PageState` so that `prev`/`next` links survive process death:

```kotlin
interface CursorPersistentPagingCache<T> {
    suspend fun save(cursor: CursorBookmark, state: PageState<T>)
    suspend fun saveAll(entries: List<Pair<CursorBookmark, PageState<T>>>)
    suspend fun load(self: Any): Pair<CursorBookmark, PageState<T>>?
    suspend fun loadAll(): List<Pair<CursorBookmark, PageState<T>>>
    suspend fun remove(self: Any)
    suspend fun removeAll(selves: List<Any>)
    suspend fun clear()
    suspend fun <R> transaction(block: suspend CursorPersistentPagingCache<T>.() -> R): R
}
```

The rest of the caching model (eviction listeners, context-window protection,
how `restart`/`release` interact with L2) matches [Caching](6.%20caching.md)
one-to-one.

CRUD tracking and L2 warm-up are available on the cursor side too ‚Äî same names,
same semantics, keyed by `CursorBookmark.self`:

```kotlin
import com.jamal_aliev.paginator.extension.warmUpFromPersistent

// Restore L1 from L2 on cold start.
val inserted = cursorPaginator.warmUpFromPersistent()

// Inspect the CRUD tracker on MutableCursorPaginator.
if (mutablePaginator.hasPendingFlush) {
    val dirty: Set<Any> = mutablePaginator.affectedSelves
    logger.info("unsaved cursors: $dirty")
}
```

See [Caching ‚Üí Warming Up L1 from L2](6.%20caching.md#warming-up-l1-from-l2)
and [Caching ‚Üí Inspecting Pending Changes](6.%20caching.md#inspecting-pending-changes)
for the offset counterpart.

## Serialization

`saveStateToJson` / `restoreStateFromJson` on both `CursorPaginator` and
`CursorPagingCore` require **two** serializers: one for the element type
`T` and one for the key type `K` (typically `String`, `Long`, `Uuid` or a
`@Serializable` data class):

```kotlin
val json: String = paginator.saveStateToJson(
    elementSerializer = Message.serializer(),
    keySerializer = String.serializer(),
)

// Process death later‚Ä¶
paginator.restoreStateFromJson(
    jsonString = json,
    elementSerializer = Message.serializer(),
    keySerializer = String.serializer(),
)
```

`contextOnly = true` is supported (saves only pages within the current
context window) ‚Äî same semantics as offset
[Serialization](5.%20serialization.md).

## Differences from offset Paginator

> Need a deep, example-driven explanation of each divergence (and the reasoning behind the
> design decision)? See the companion doc
> [Paginator vs CursorPaginator](14.%20paginator-vs-cursor.md). The table below is a
> quick-reference summary.

| Concern                               | Offset `Paginator`                                                                                                         | `CursorPaginator`                                     |
|---------------------------------------|----------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------|
| Page address                          | `Int` (>= 1)                                                                                                               | `CursorBookmark.self: Any`                            |
| End-of-feed signal                    | `finalPage` (upper bound) + explicit                                                                                       | `next == null` / `prev == null` on the edge bookmark  |
| End-of-feed exception                 | `FinalPageExceededException`                                                                                               | `EndOfCursorFeedException` (FORWARD / BACKWARD)       |
| Load guard exception                  | `LoadGuardedException(page: Int)`                                                                                          | `CursorLoadGuardedException(attemptedCursor)`         |
| Initial "what is page 1?"             | Implicit ‚Äî always `BookmarkInt(1)`                                                                                         | `initialCursor` (optional) or server-returned first   |
| Random-access jump                    | Yes (`jump(BookmarkInt(42))`)                                                                                              | Only to a cursor you already know                     |
| `resize(capacity)`                    | Redistributes data across pages                                                                                            | **Unsupported** ‚Äî cursors can't be renumbered         |
| `near` / `far` / `gap` on `PageState` | Supported                                                                                                                  | **Unavailable** ‚Äî no numeric distance between cursors |
| `removeState` ripple effect           | Collapses downstream page numbers                                                                                          | Re-links neighbours only ‚Äî O(1)                       |
| `addAllElements` overflow cascade     | Walks `page + 1`                                                                                                           | Walks `cursor.next`; optionally spawns a new tail     |
| Bookmarks default content             | `[BookmarkInt(1)]`                                                                                                         | `[]` (cursor keys are opaque and server-provided)     |
| `elementAtOrNull(globalIndex)`        | O(1) via page math                                                                                                         | O(N) ‚Äî intentional trade-off                          |
| Common concerns                       | `PageState`, `uiState`, `Interweaver`, logger, `transaction`, capacity, prefetch, lock flags, CRUD surface ‚Äî all identical |
