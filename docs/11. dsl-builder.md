## Why a DSL?

Constructing a `MutablePaginator` manually requires setting up `PagingCore`, wiring optional
caches, then post-configuring properties one by one:

```kotlin
val core = PagingCore<User>(
    cache = LruPagingCache(maxSize = 50) + TtlPagingCache(ttl = 5.minutes),
    persistentCache = roomCache,
    initialCapacity = 20,
)
val paginator = MutablePaginator(core = core, load = { page -> api.getUsers(page) }).apply {
    finalPage = 100
    recyclingBookmark = true
    logger = PrintPaginatorLogger()
    bookmarks.clear()
    bookmarks.addAll(listOf(BookmarkInt(1), BookmarkInt(10), BookmarkInt(50)))
    core.initializerEmptyPage = { page, data, meta -> LabeledEmptyPage(page, data, meta) }
}
```

The DSL collapses this into a single declarative block:

```kotlin
val paginator = mutablePaginator<User>(capacity = 20) {
    load { page -> api.getUsers(page) }
    cache = LruPagingCache(maxSize = 50) + TtlPagingCache(ttl = 5.minutes)
    persistentCache = roomCache
    finalPage = 100
    recyclingBookmark = true
    logger = PrintPaginatorLogger()
    bookmarks(1, 10, 50)
    initializers {
        empty { page, data, meta -> LabeledEmptyPage(page, data, meta) }
    }
}
```

## Two entry points: `paginator` vs `mutablePaginator`

The DSL exposes **two** top-level functions, mirroring the
[`Paginator` / `MutablePaginator` split](1.%20core-concepts.md#paginator-vs-mutablepaginator):

| DSL                         | Returns               | Use when                                                                            |
|-----------------------------|-----------------------|-------------------------------------------------------------------------------------|
| `paginator<T> { … }`        | `Paginator<T>`        | The call site only needs navigation, refresh, and snapshot flow access.             |
| `mutablePaginator<T> { … }` | `MutablePaginator<T>` | The call site needs element-level CRUD (`addElement`, `removeElement`, `flush`, …). |

Returning a read-only `Paginator` from `paginator { }` is a **deliberate compile-time guarantee**
— downstream code cannot accidentally call mutating methods. If you need mutations, opt in by
using `mutablePaginator { }`.

```kotlin
// Read-only — `setElement`/`removeElement` are not in scope at the call site
val users: Paginator<User> = paginator(capacity = 20) {
    load { page -> api.getUsers(page) }
    finalPage = 100
}

// Full CRUD — caller can `users.setElement(...)`, `users.flush()`, etc.
val users: MutablePaginator<User> = mutablePaginator(capacity = 20) {
    load { page -> api.getUsers(page) }
}
```

Both DSLs share the **same set of configuration knobs** — only the build product differs.

## Minimal example

The `load { }` block is the only required call. Everything else has sensible defaults
(see [Available knobs](#available-knobs)).

```kotlin
val users = mutablePaginator<User> {
    load { page -> api.getUsers(page) }
}
```

If `load { }` is omitted, the builder throws `IllegalStateException` at build time.

## Full example

```kotlin
val articles: MutablePaginator<Article> = mutablePaginator(capacity = 20) {
    load { page ->
        val response = api.getArticles(page)
        LoadResult(response.items, ArticleMetadata(response.totalPages))
    }

    // Cache: stack eviction strategies with the `+` operator.
    cache = LruPagingCache(maxSize = 50) + TtlPagingCache(ttl = 5.minutes)
    persistentCache = roomArticleCache

    // Navigation
    finalPage = 100
    bookmarks(1, 25, 50, 75, 100)
    recyclingBookmark = true

    // Diagnostics
    logger = PrintPaginatorLogger(minLevel = LogLevel.DEBUG)

    // Custom PageState factories — any factory you don't override keeps its default.
    initializers {
        empty { page, data, meta -> LabeledEmptyPage(page, data, meta) }
        error { e, page, data, meta -> RetryableErrorPage(e, page, data, meta) }
    }
}
```

## Available knobs

### `load { }` (required)

The suspending lambda used to fetch a page from the source. Same signature as the `load`
parameter on the `Paginator`/`MutablePaginator` constructors.

```kotlin
load { page -> LoadResult(api.getItems(page)) }
```

### Capacity

Pass `capacity` to the top-level builder function (not as a property inside the block):

```kotlin
mutablePaginator<User>(capacity = 20) { … }
```

Default: `PagingCore.DEFAULT_CAPACITY` (20). Use `PagingCore.UNLIMITED_CAPACITY` (0) to disable
capacity checks (
see [Capacity & Incomplete Pages](1.%20core-concepts.md#capacity--incomplete-pages)).

### `cache` and `persistentCache`

```kotlin
cache = LruPagingCache(maxSize = 50) + TtlPagingCache(ttl = 5.minutes)
persistentCache = myRoomBackedCache
```

`cache` defaults to `DefaultPagingCache()` (no eviction). `persistentCache` defaults to `null`
(no L2). See [Caching](6.%20caching.md) for the full eviction model and how to compose strategies
with the `+` operator.

### `logger`, `finalPage`, `recyclingBookmark`

Settable as properties inside the block; they map 1:1 to the corresponding fields on
`Paginator`.

```kotlin
logger = PrintPaginatorLogger()
finalPage = 100
recyclingBookmark = true
```

### `bookmarks(...)`

Replaces the default bookmark list (a single `BookmarkInt(1)`). Two overloads:

```kotlin
// Vararg of page numbers (wrapped in BookmarkInt)
bookmarks(1, 10, 50, 100)

// Pre-built Bookmark instances (custom subtypes welcome)
bookmarks(listOf(BookmarkInt(1), LabeledBookmark("home")))
```

Pass nothing (`bookmarks()`) to clear the list entirely.

### `initializers { }`

Sub-builder for overriding the four `PageState` factories on the underlying `PagingCore`. Any
factory you don't call keeps its default.

```kotlin
initializers {
    progress { page, data, meta -> SkeletonProgressPage(page, data, meta) }
    success { page, data, meta -> EnrichedSuccessPage(page, data, meta) }
    empty { page, data, meta -> LabeledEmptyPage(page, data, meta) }
    error { e, page, data, meta -> RetryableErrorPage(e, page, data, meta) }
}
```

The block can be invoked multiple times — later calls **merge** with earlier ones (only the
slots they actually set are overwritten). Equivalent to manually assigning
`paginator.core.initializerXxxPage = { … }` after construction.

See [Custom PageState Subclasses](4.%20elements.md#custom-pagestate-subclasses) for the full
metadata-propagation pattern.

## Builder hierarchy

The two builders share their entire configuration surface through a sealed base class:

```
BasePaginatorBuilder<T>          ← sealed, holds all the shared knobs
├── PaginatorBuilder<T>           → build(): Paginator<T>
└── MutablePaginatorBuilder<T>    → build(): MutablePaginator<T>
```

`@PaginatorDsl` is applied as a `@DslMarker` so nested DSL receivers (e.g. `initializers { }`)
cannot accidentally call methods on the outer builder.

The hierarchy is **sealed** — you cannot register your own builder. If you need a custom subclass
of `MutablePaginator`, instantiate it directly rather than via the DSL.

---

## Migration from manual construction

Both DSLs are **purely additive** — manual construction via `MutablePaginator(core, load)` and
`Paginator(core, load)` continues to work exactly as before. Adopt the DSL one call site at a time.

| Manual                                                  | DSL                                              |
|---------------------------------------------------------|--------------------------------------------------|
| `Paginator(core, load).apply { … }`                     | `paginator<T>(capacity) { load { … } … }`        |
| `MutablePaginator(core, load).apply { … }`              | `mutablePaginator<T>(capacity) { load { … } … }` |
| `core.initializerEmptyPage = { … }`                     | `initializers { empty { … } }`                   |
| `paginator.bookmarks.clear(); paginator.bookmarks += b` | `bookmarks(b1, b2, …)` or `bookmarks(listOf(…))` |
