# Prefetch (Auto-Pagination on Scroll)

[← Back to README](../README.md)

`PaginatorPrefetchController` watches the visible window of a list and triggers
`goNextPage()` / `goPreviousPage()` **before** the user reaches the edge, so new pages are
fetched in advance and the list scrolls without "load more" buttons.

The controller is **platform-agnostic** — it consumes visibility via a single
`onScroll(firstVisibleIndex, lastVisibleIndex, totalItemCount)` call, which means the same
core logic powers RecyclerView, LazyColumn / Grid / StaggeredGrid, UITableView, or any other
list. Reversed layouts (`reverseLayout = true`, chats) work without special configuration.

> **Cursor counterpart:** `CursorPaginator.prefetchController(...)` returns a
> `CursorPaginatorPrefetchController` with identical configuration and lifecycle. The only
> difference is the edge check — forward fires when `endContextCursor?.next != null`,
> backward when `startContextCursor?.prev != null`. Every binding shown below has a
> cursor-paginator overload with the same parameters; substitute `CursorLoadGuard` for
> `PageLoadGuard` and a `CursorBookmark` for the page number. See
> [Cursor-Based Pagination → Prefetch](13.%20cursor-pagination.md#prefetch).

## How to read this guide

This document is intentionally structured **from the lowest level to the highest**, so you
can see exactly *what each layer of automation buys you*. Five layers, all resolving to the
same `PaginatorPrefetchController` underneath:

| #           | What you write                                                                                | What is automated                                                    |
|-------------|-----------------------------------------------------------------------------------------------|----------------------------------------------------------------------|
| **Layer 0** | Scroll listener + `paginator.goNextPage()` by hand                                            | Nothing. Pure domain API.                                            |
| **Layer 1** | Scroll listener + `controller.onScroll(...)` by hand                                          | Calibration, direction tracking, locks, `loadGuard`, `finalPage`.    |
| **Layer 2** | `rememberPrefetchController` + `BindToLazyList` / `prefetchController` + `bindToRecyclerView` | Scroll & layout listeners, header offset, dedup, lifecycle.          |
| **Layer 3** | `PrefetchOnScroll` / `bindPrefetchToRecyclerView`                                             | Layer 2 + controller construction in one call.                       |
| **Layer 4** | `rememberPaginated` + `paginated { … }` / `bindPaginated`                                     | Layer 3 + automatic `dataItemCount` / `headerCount` / `footerCount`. |

You can stop at any layer that matches the amount of control you need. **Switching layers
is a refactor, not a rewrite** — the underlying contract is the same: `totalItemCount` is
*data items only*, headers / footers / append-loaders are excluded. Only the boilerplate
shrinks.

If you came for a recipe, jump to [Mini quick start](#mini-quick-start-top-layer) and
[Layer 4](#layer-4--auto-everything-dsl--bindpaginated). If you came to understand what
the library is doing for you, read top to bottom.

## Mini quick start (top layer)

For the impatient — top layer in both UIs. See
[Layer 4](#layer-4--auto-everything-dsl--bindpaginated) for the full contract.

**Compose** (`paginator-compose`):

```kotlin
val listState = rememberLazyListState()
val paged = paginator.rememberPaginated(state = listState)

LazyColumn(state = listState) {
    paginated(paged) {
        header { StickyTitle() }
        items(uiState.items, key = { it.id }) { Row(it) }
        appendIndicator { AppendIndicator(uiState.appendState) }
    }
}
```

**Android View** (`paginator-view`):

```kotlin
binding.recyclerView.adapter = ConcatAdapter(headerAdapter, dataAdapter, appendIndicatorAdapter)
binding.recyclerView.layoutManager = LinearLayoutManager(context)

val paged = paginator.bindPaginated(
    recyclerView = binding.recyclerView,
    lifecycleOwner = viewLifecycleOwner,
    headerCount = { headerAdapter.itemCount },
    footerCount = { appendIndicatorAdapter.itemCount },
)
```

## Layer 0 — Manual edge detection (no controller)

The most primitive integration: no `PaginatorPrefetchController`, no UI artifacts. You
listen to scroll yourself and call `paginator.goNextPage()` / `paginator.goPreviousPage()`
when the visible window approaches the edge.

This layer exists in this guide for one reason — **to make explicit what every higher
layer does for you**. You almost never want to write it in production.

### Compose

```kotlin
val listState = rememberLazyListState()
val coroutineScope = rememberCoroutineScope()

// Naive edge detection — fires every time visibility changes.
LaunchedEffect(listState, uiState.items.size) {
    snapshotFlow {
        val layoutInfo = listState.layoutInfo
        val lastVisibleIndex = layoutInfo.visibleItemsInfo.lastOrNull()?.index ?: -1
        lastVisibleIndex to uiState.items.size
    }
        .distinctUntilChanged()
        .collect { (lastVisibleIndex, totalDataItems) ->
            // Manually mirror what the controller does — and discover all the edge cases.
            if (totalDataItems > 0 && lastVisibleIndex >= 0 && lastVisibleIndex >= totalDataItems - PREFETCH_DISTANCE) {
                coroutineScope.launch { paginator.goNextPage() }
            }
        }
}

LazyColumn(state = listState) {
    items(uiState.items, key = { it.id }) { Row(it) }
}
```

### Android View

```kotlin
binding.recyclerView.addOnScrollListener(object : RecyclerView.OnScrollListener() {
    override fun onScrolled(recyclerView: RecyclerView, dx: Int, dy: Int) {
        val layoutManager = recyclerView.layoutManager as LinearLayoutManager
        val lastVisibleIndex = layoutManager.findLastVisibleItemPosition()
        val totalDataItems = uiState.items.size
        if (totalDataItems > 0 && lastVisibleIndex >= 0 && lastVisibleIndex >= totalDataItems - PREFETCH_DISTANCE) {
            viewLifecycleOwner.lifecycleScope.launch { paginator.goNextPage() }
        }
    }
})
```

### Everything you have to handle yourself at this layer

The snippet above is wrong in at least seven ways the moment you ship it. Fixing each by
hand is exactly what `PaginatorPrefetchController` does for you in Layer 1+:

| Problem                                                                                                                                                      | Layer 1+ behaviour                                                                                                                |
|--------------------------------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------|
| The first emission is a false positive: the list mounts at the edge and immediately fires `goNextPage()`.                                                    | First `onScroll` is **calibration only**, never triggers a load.                                                                  |
| Re-firing on every recomposition / scroll tick → multiple in-flight `goNextPage()` calls fighting each other.                                                | Forward fires only when `lastVisibleIndex` *increased* since the previous call; in-flight job is tracked.                         |
| Backward prefetch (chats, jump-to-anchor) needs symmetric logic against `firstVisibleIndex`.                                                                 | `enableBackwardPrefetch = true` adds it.                                                                                          |
| `finalPage` reached, `lockGoNextPage`, `loadGuard` are not respected.                                                                                        | Controller checks all three before dispatching.                                                                                   |
| First page is **shorter than the viewport** → `OnScrollListener` never fires, append-pagination stalls forever.                                              | Layer 2+ View binding installs an `OnLayoutChangeListener` (Compose: `snapshotFlow { layoutInfo }`) that drives a synthetic call. |
| If you render headers / footers / append-loader, `total` and `lastVisible` are inflated → off-by-one against `prefetchDistance`.                             | Controller's contract is **data-only** indices; bindings subtract `headerCount` and clamp to `dataItemCount - 1` for you.         |
| Errors during `goNextPage()` need an `onPrefetchError` callback, otherwise they hit your global crash handler or get silently swallowed by `coroutineScope`. | Controller surfaces them via `onPrefetchError` / [`PrefetchErrorChannel`](#reactive-errors-prefetcherrorchannel).                 |

Move to Layer 1 the moment any of these matter — i.e., immediately, in any real screen.

## Layer 1 — Controller, manual `onScroll`

You construct a `PaginatorPrefetchController` yourself (or via the
`paginator.prefetchController(scope = ...)` extension), wire your own scroll listener, and
forward visibility into `controller.onScroll(first, last, total)`.

This layer pulls in **calibration, direction tracking, lock / `finalPage` / `loadGuard`
checks**, and serializes prefetch jobs — but you still write the listener and pass
data-only indices yourself.

### Pure-Kotlin construction

```kotlin
val prefetch = paginator.prefetchController(
    scope = viewModelScope,
    prefetchDistance = 10,
    enableBackwardPrefetch = true,
)
```

### Compose

```kotlin
LaunchedEffect(listState, prefetch) {
    snapshotFlow {
        val layoutInfo = listState.layoutInfo
        Triple(
            listState.firstVisibleItemIndex,
            layoutInfo.visibleItemsInfo.lastOrNull()?.index ?: 0,
            uiState.items.size,    // data items only — no headers / footers / loader
        )
    }.collect { (firstVisibleIndex, lastVisibleIndex, totalItemCount) ->
        prefetch.onScroll(firstVisibleIndex, lastVisibleIndex, totalItemCount)
    }
}
```

### Android View

```kotlin
binding.recyclerView.addOnScrollListener(object : RecyclerView.OnScrollListener() {
    override fun onScrolled(recyclerView: RecyclerView, dx: Int, dy: Int) {
        val layoutManager = recyclerView.layoutManager as LinearLayoutManager
        prefetch.onScroll(
            firstVisibleIndex = layoutManager.findFirstVisibleItemPosition(),
            lastVisibleIndex = layoutManager.findLastVisibleItemPosition(),
            totalItemCount = uiState.items.size,
        )
    }
})
```

### What you still own at this layer

- **`totalItemCount` must be data-only.** If your adapter renders headers / footers /
  dividers, subtract them yourself before calling `onScroll`.
- **First-page-shorter-than-viewport.** `OnScrollListener` never fires when nothing
  scrolls. You have to either install an `OnLayoutChangeListener` to dispatch one
  synthetic call after layout, or — in Compose — make the `snapshotFlow` block depend on
  `listState.layoutInfo` so it re-emits on layout passes (the snippet above already does
  the latter implicitly via `visibleItemsInfo`).
- **Listener teardown.** Remove the `OnScrollListener` on `ON_DESTROY` yourself.
- **Reverse layouts.** The controller works on data indices, not visual positions — pass
  the same indices you would for a normal list. In a chat (`reverseLayout = true`),
  forward prefetch fires when the user scrolls **upward** toward older messages.

Move to Layer 2 to delete the listener boilerplate and the layout-change handling.

## Layer 2 — Controller + UI binding

You still build the controller (so you can stash it in a `ViewModel`, share it across
fragments, hot-mutate `enabled` / `prefetchDistance`, etc.), but the scroll & layout
listeners and the data-only-index math are installed for you by a binding.

This is the layer to use when **the controller's lifetime is wider than the screen** —
typically a `viewModelScope` controller bound to a transient `Fragment` view.

> **Heads up — extra dependency required from here on.** Layer 0 and Layer 1 use only
> the core `paginator` artifact. Starting with Layer 2, the bindings live in dedicated
> UI artifacts. The recommended way is to import the **BOM** once and then declare the
> artifacts you need without versions — that way `paginator`, `paginator-compose`, and
> `paginator-view` cannot drift on your classpath:
>
> ```kotlin
> dependencies {
>     implementation(platform("io.github.jamal-wia:paginator-bom:8.7.0"))
>
>     // core
>     implementation("io.github.jamal-wia:paginator")
>
>     // Compose binding — BindToLazyList / PrefetchOnScroll / rememberPaginated
>     implementation("io.github.jamal-wia:paginator-compose")
>
>     // Android View binding — bindToRecyclerView / bindPrefetchToRecyclerView / bindPaginated
>     implementation("io.github.jamal-wia:paginator-view")
> }
> ```
>
> Pull in only the UI artifact you actually use — they're independent. The BOM only pins
> *Paginator* artifacts; it does not constrain Compose, Kotlin, AndroidX, or anything
> else on your classpath.

### Compose — `BindToLazyList`

```kotlin
val listState = rememberLazyListState()

val prefetch = paginator.rememberPrefetchController(
    prefetchDistance = 10,
    enableBackwardPrefetch = true,
)

prefetch.BindToLazyList(
    listState = listState,
    dataItemCount = uiState.items.size,
    headerCount = 1,
    footerCount = 1,
)

LazyColumn(state = listState) {
    item { Header() }
    items(uiState.items, key = { it.id }) { Row(it) }
    item { AppendIndicator(uiState.appendState) }
}
```

`BindToLazyGrid` / `BindToLazyStaggeredGrid` are the equivalents on
`PaginatorPrefetchController` and `CursorPaginatorPrefetchController`.

Re-emitting `dataItemCount` / `headerCount` / `footerCount` across recompositions does
**not** restart the underlying flow or re-calibrate — values are read fresh on every
emission. The flow restarts only when the controller identity, the lazy-state instance,
or [`restartKey`](#re-calibration-reset-vs-recalibrate-vs-restartkey) changes.

### Android View — `bindToRecyclerView`

```kotlin
class FeedViewModel : ViewModel() {
    val paginator: Paginator<Item> = ...
    val prefetch = paginator.prefetchController(
        scope = viewModelScope,
        prefetchDistance = 10,
    )
}

class FeedFragment : Fragment() {
    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        viewModel.prefetch.bindToRecyclerView(
            recyclerView = binding.recyclerView,
            lifecycleOwner = viewLifecycleOwner,
            dataItemCount = { adapter.itemCount },
            headerCount = { headerAdapter.itemCount },
            footerCount = { appendAdapter.itemCount },
        )
    }
}
```

`bindToRecyclerView` installs both an `OnScrollListener` **and** an
`OnLayoutChangeListener` — the latter is what handles "first page shorter than the
viewport" without requiring user scroll. Both listeners are removed on `ON_DESTROY`. The
`dataItemCount` / `headerCount` / `footerCount` lambdas are read on **every dispatch**, so
dynamic header/footer counts work without any state-management ceremony.

> The View `bindToRecyclerView` does **not** cancel the controller — controllers are
> typically scoped to a `ViewModel` and outlive the view. Use the lifecycle-aware
> `paginator.prefetchController(lifecycleOwner = ...)` from `paginator-view` to also tie
> controller lifetime to the fragment.

### What this layer buys over Layer 1

- Scroll & layout listeners installed and torn down for you.
- Header offset and `lastVisibleIndex` clamping handled — you can pass *adapter* indices
  through the lambdas and the binding converts to data-only.
- Dedup: a layout pass that produces the same indices as the previous scroll event is a
  no-op.
- Optional throttling via [
  `PrefetchOptions.scrollSampleMillis`](#configuration-prefetchoptions-and-controller-properties).

You still pass `dataItemCount` / `headerCount` / `footerCount` yourself.

## Layer 3 — One-call helper (controller + binding bundled)

Same surface as Layer 2, except the controller is **created and bound in a single call**.
Use this when prefetch lives for the lifetime of the screen and the controller does not
need to be referenced from anywhere else (no `ViewModel` field, no "Pause prefetch"
button outside the screen).

### Compose — `PrefetchOnScroll`

```kotlin
val listState = rememberLazyListState()

paginator.PrefetchOnScroll(
    state = listState,
    dataItemCount = uiState.items.size,
    headerCount = 1,
    footerCount = 1,
    options = PrefetchOptions(prefetchDistance = 10, enableBackwardPrefetch = true),
)
```

Overloads exist for `LazyListState`, `LazyGridState`, `LazyStaggeredGridState` ×
`Paginator`, `CursorPaginator`. Returns the constructed controller if you need the
reference (e.g., to flip `enabled = false` from a button on the same screen).

### Android View — `bindPrefetchToRecyclerView`

```kotlin
val handle = paginator.bindPrefetchToRecyclerView(
    recyclerView = binding.recyclerView,
    lifecycleOwner = viewLifecycleOwner,
    dataItemCount = { adapter.itemCount },
    headerCount = 1,
    footerCount = 1,
    options = PrefetchOptions(prefetchDistance = 10, enableBackwardPrefetch = true),
)

// handle.controller — direct reference to the PaginatorPrefetchController
// handle.recalibrate() / handle.unbind() — ScrollBinding methods (delegated)
```

### What this layer buys over Layer 2

- One call instead of two (controller construction + binding).
- The controller is created against the right scope automatically (`rememberCoroutineScope`
  in Compose, `lifecycleOwner.lifecycleScope` in View).
- `cancelOnDispose` is honored — the controller is cancelled on teardown by default. See
  [Lifecycle](#lifecycle-and-canceldispose).

`dataItemCount` / `headerCount` / `footerCount` are still your responsibility.

## Layer 4 — Auto-everything (DSL / `bindPaginated`)

The top layer also takes over `dataItemCount` / `headerCount` / `footerCount`, so you
write **zero numbers** related to prefetch. This is what the [Mini quick
start](#mini-quick-start-top-layer) at the top shows.

### Compose — `rememberPaginated` + `paginated { … }`

```kotlin
val listState = rememberLazyListState()
val paged = paginator.rememberPaginated(state = listState)

LazyColumn(state = listState) {
    paginated(paged) {
        header { StickyTitle() }
        items(uiState.items, key = { it.id }) { Row(it) }
        appendIndicator { AppendIndicator(uiState.appendState) }
    }
}
```

What `rememberPaginated` + `paginated { … }` give you for free:

- **`dataItemCount`** is derived from `paginator.uiState` — only `Content.items.size`
  propagates through a primitive `MutableIntState`. Loading / error state flips do **not**
  trigger recomposition of the binding.
- **`headerCount` / `footerCount`** are tallied by `header { }` / `appendIndicator { }`
  inside the DSL block on every composition, so the classic off-by-one bug around the
  append-loader is impossible — even if the loader appears or disappears conditionally.
- The `paginated` scope behaves like a normal `LazyListScope`; `item`, `items`,
  `itemsIndexed`, `stickyHeader` work unchanged.

The same DSL is available for grids and staggered grids — `LazyVerticalGrid`,
`LazyHorizontalGrid`, `LazyVerticalStaggeredGrid` all have a matching `rememberPaginated` /
`paginated { … }` pair. `header { }` spans the full row by default
(`StaggeredGridItemSpan.FullLine` for staggered).

The holder returned by `rememberPaginated` exposes the underlying controller as
`paged.controller` — useful when you want to invoke `cancel()`, `reset()`, or flip
`enabled = false` from a button.

### Android View — `bindPaginated`

```kotlin
binding.recyclerView.adapter = ConcatAdapter(headerAdapter, dataAdapter, appendIndicatorAdapter)
binding.recyclerView.layoutManager = LinearLayoutManager(context)

val paged = paginator.bindPaginated(
    recyclerView = binding.recyclerView,
    lifecycleOwner = viewLifecycleOwner,
    headerCount = { headerAdapter.itemCount },
    footerCount = { appendIndicatorAdapter.itemCount },
)
```

What `bindPaginated` gives you for free:

- **`dataItemCount` is auto-tracked** from `paginator.uiState` (only `Content.items.size`
  triggers re-dispatch — transient loading / error states do not).
- **The controller is created and lifecycle-scoped** for you; on `ON_DESTROY` it cancels
  per [`PrefetchOptions.cancelOnDispose`](#configuration-prefetchoptions-and-controller-properties).
- Both `OnScrollListener` and `OnLayoutChangeListener` are installed and removed
  automatically. The layout listener is what handles "**first page is shorter than the
  viewport**" — without it, append-pagination would stall on partial first pages.
- The returned `PrefetchBinding<PaginatorPrefetchController<T>>` exposes `.controller`
  (for hot-update mutation, `cancel()`, etc.) and the `ScrollBinding` interface
  (`.recalibrate()` after a manual refresh / jump, `.unbind()` to detach early).

Supported `LayoutManager`s: `LinearLayoutManager`, `GridLayoutManager`,
`StaggeredGridLayoutManager`. The binding always passes `totalItemCount` as **data items
only** — headers, footers, append loaders, and dividers are excluded regardless of how you
express them in the adapter.

### What this layer buys over Layer 3

- No `dataItemCount` to pass — the binding subscribes to `paginator.uiState` itself.
- No `headerCount` / `footerCount` numbers in Compose — the DSL counts them.
- In View, `headerCount` / `footerCount` are still passed, but as **lambdas reading from
  your adapters** instead of magic constants — there is no number to keep in sync with
  the layout.

This is the default. Drop down only when you need something this layer can't do (a
controller shared across screens, custom dispatch logic, manual integration without the
artifacts).

## Integration layers — overview table

The same matrix as the intro, with concrete entry points filled in. Use it as a
cheat-sheet when you've decided which layer fits.

| Layer       | Compose entry point                                                | View entry point                                   | `dataItemCount` | `headerCount` / `footerCount` |
|-------------|--------------------------------------------------------------------|----------------------------------------------------|-----------------|-------------------------------|
| **Layer 0** | hand-rolled `snapshotFlow` + `goNextPage()`                        | hand-rolled `OnScrollListener`                     | manual          | manual                        |
| **Layer 1** | hand-rolled `snapshotFlow` + `controller.onScroll(...)`            | hand-rolled listener + `onScroll(...)`             | manual          | manual                        |
| **Layer 2** | `rememberPrefetchController` + `BindToLazyList/Grid/StaggeredGrid` | `prefetchController(scope)` + `bindToRecyclerView` | manual          | manual                        |
| **Layer 3** | `PrefetchOnScroll`                                                 | `bindPrefetchToRecyclerView`                       | manual          | manual                        |
| **Layer 4** | `rememberPaginated` + `paginated { … }` DSL                        | `bindPaginated`                                    | **auto**        | **auto**                      |

Every layer honors the same contract: the `totalItemCount` reaching the controller is
**data items only** — headers, footers, append loaders, and dividers are never counted.
Switching layers is a refactor, not a rewrite, because all five resolve to the same
`PaginatorPrefetchController` underneath.

## Configuration (`PrefetchOptions` and controller properties)

`PrefetchOptions` is a single immutable bag accepted by every Layer 2+ binding:

```kotlin
val options = PrefetchOptions(
    prefetchDistance = 10,
    enableBackwardPrefetch = true,
    silentlyLoading = true,
    silentlyResult = false,
    enabled = true,
    cancelOnDispose = true,
    scrollSampleMillis = 0L,
)
```

Because `PrefetchOptions` is a `data class` with all `val` primitive fields, the Compose
compiler treats it as **stable** — passing the same instance through recompositions does
not invalidate downstream skipping. Hoist it (`remember { PrefetchOptions(...) }`) to share
configuration across screens.

### Fields shared between binding and controller

| Field                    | Default                         | Description                                                                                                                                                               |
|--------------------------|---------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `prefetchDistance`       | `DefaultPrefetchDistance` (`6`) | Number of **data items** from the edge that trigger prefetch. Must be `> 0`. In a grid this is **cells, not rows** — for `GridCells.Fixed(3)`, distance `6` means 2 rows. |
| `enableBackwardPrefetch` | `false`                         | Also prefetch toward the start of the list when the user scrolls back.                                                                                                    |
| `silentlyLoading`        | `true`                          | Suppress the `ProgressPage` snapshot during prefetch loading (avoids a spinner for a page the user hasn't reached yet).                                                   |
| `silentlyResult`         | `false`                         | Suppress the snapshot when the prefetched page lands. Usually `false` so the UI picks up the data immediately.                                                            |
| `enabled`                | `true`                          | Master switch. `false` makes `onScroll(...)` a no-op (positions are still tracked so re-enabling resumes correctly).                                                      |
| `cancelOnDispose`        | `true`                          | Cancel in-flight prefetch on Compose `onDispose` / View `ON_DESTROY`. See [Lifecycle](#lifecycle-and-canceldispose) for the interaction with the controller's scope.      |
| `scrollSampleMillis`     | `0` (no throttling)             | When `> 0`, throttle the scroll/layout signal via `Flow.sample` so only the latest emission per window reaches the controller. See note below.                            |

`scrollSampleMillis` adds up to one window of latency before prefetch fires. On fast
scrolls and slow devices, the user can cross the prefetch boundary inside that window. If
you set it, increase `prefetchDistance` to absorb the throttle (`distance ≥ items_scrolled
per window + safety_margin`). `16L` (one frame) is a safe starting point.

### Fields available only on the controller

These are **not** in `PrefetchOptions` because they're either non-primitive (would break
Compose stability) or rarely overridden. Set them on the controller after construction —
or pass them to `paginator.prefetchController(...)` directly when integrating manually.

| Property            | Default                                  | Description                                                                                                                                                           |
|---------------------|------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `loadGuard`         | `{ _, _ -> true }`                       | Guard callback forwarded to `goNextPage` / `goPreviousPage`. Prefer `PageLoadGuard` (below).                                                                          |
| `onPrefetchError`   | `null`                                   | Invoked when a prefetch fails (excluding `CancellationException`). `null` → silently ignored. Prefer [`PrefetchErrorChannel`](#reactive-errors-prefetcherrorchannel). |
| `enableCacheFlow`   | `paginator.core.enableCacheFlow`         | Forwarded to navigation functions; override only if you intentionally diverge from the paginator's default.                                                           |
| `initProgressState` | `paginator.core.initializerProgressPage` | Factory for `PageState.ProgressPage` during prefetch.                                                                                                                 |
| `initSuccessState`  | `paginator.core.initializerSuccessPage`  | Factory for `PageState.SuccessPage` on a successful prefetch.                                                                                                         |
| `initErrorState`    | `paginator.core.initializerErrorPage`    | Factory for `PageState.ErrorPage` on a failed prefetch.                                                                                                               |

All controller properties are mutable at runtime — flip `enabled`, change `prefetchDistance`,
swap `loadGuard`, etc., from anywhere with a controller reference (top-layer holder
exposes `paged.controller`; lower layers keep the controller you constructed).

### `onScroll` input contract

The controller silently no-ops in the following cases — useful when debugging "why didn't
prefetch fire?":

- `totalItemCount <= 0` → ignored.
- `firstVisibleIndex < 0` or `lastVisibleIndex < 0` → ignored.
- `lastVisibleIndex >= totalItemCount` → clamped to `totalItemCount - 1`.
- The very first call is **calibration only** — it records positions and never triggers
  prefetch. This prevents a false positive when the list first appears.
- Forward prefetch fires only on **scrolling forward** (`lastVisibleIndex` increased
  since the previous call). Backward fires only on **scrolling backward**. If the user
  lands at the edge and stays still, prefetch will not fire until they scroll.
- Forward additionally requires `endContextPage < finalPage` and not `lockGoNextPage`;
  backward requires `startContextPage > 1` and not `lockGoPreviousPage`.

## Reactive errors (`PrefetchErrorChannel`)

A stable `StateFlow`-backed channel that captures the latest prefetch error. `PrefetchErrorChannel`
is `@Stable` and captures `onError` once, so passing it through recompositions does not
re-create the controller.

### Compose

```kotlin
val errors = rememberPrefetchErrorChannel()

paginator.PrefetchOnScroll(
    state = listState,
    dataItemCount = uiState.items.size,
    onPrefetchError = errors.onError,   // stable reference, safe across recompositions
)

errors.current?.let { error ->
    LaunchedEffect(error) {
        snackbar.showSnackbar(error.message ?: "Prefetch failed")
        errors.consume()
    }
}
```

### View

The same `PrefetchErrorChannel` works in Fragments — just collect the `errors` `StateFlow`
instead of reading `current`:

```kotlin
private val errors = PrefetchErrorChannel()

override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    paginator.bindPaginated(
        recyclerView = binding.recyclerView,
        lifecycleOwner = viewLifecycleOwner,
        onPrefetchError = errors.onError,
    )

    viewLifecycleOwner.lifecycleScope.launch {
        viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            errors.errors.collect { error ->
                if (error != null) {
                    Snackbar.make(view, error.message ?: "Prefetch failed", LENGTH_SHORT).show()
                    errors.consume()
                }
            }
        }
    }
}
```

`errors.current` (snapshot) and `errors.errors.collect { … }` (stream) are equivalent reads
of the same `StateFlow<Exception?>` — pick whichever fits the consumer. Call `consume()`
to clear the latest error.

## Stable load guards

`PageLoadGuard` / `CursorLoadGuard` are `fun interface` types whose instances Compose
treats as stable, unlike raw lambdas. Use them to keep guard identity stable across
recompositions and reuse:

```kotlin
private val onlyAfterFirstPage = PageLoadGuard<Item> { page, _ -> page > 1 }

// Compose
val paged = paginator.rememberPaginated(state = listState, loadGuard = onlyAfterFirstPage)

// View
paginator.bindPaginated(
    recyclerView = binding.recyclerView,
    lifecycleOwner = viewLifecycleOwner,
    loadGuard = onlyAfterFirstPage,
)
```

Use the cached `PageLoadGuard.allowAll()` / `CursorLoadGuard.allowAll()` singletons when
no filtering is needed (this is the default).

## Re-calibration: `reset()` vs `recalibrate()` vs `restartKey`

Three mechanisms — three different scopes. Pick the one whose scope matches what changed.

| Mechanism               | Scope                   | In-flight jobs | Calibration | Scroll-observation flow | Use when                                                                                                                                          |
|-------------------------|-------------------------|----------------|-------------|-------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------|
| `controller.reset()`    | Controller              | Cancelled      | Cleared     | Untouched               | List contents change entirely (`paginator.release()`, large `resize()`).                                                                          |
| `binding.recalibrate()` | Binding                 | Untouched      | Cleared     | Untouched               | After a manual `refresh()` / `jump()` when you want the next event treated as fresh calibration.                                                  |
| `restartKey` (Compose)  | Scroll-observation flow | Untouched      | Cleared     | Restarted               | You need the binding's coroutine torn down and re-launched (e.g. you swapped the controller's enabled state and want to re-observe from scratch). |

Notes:

- `bindPaginated` calls `recalibrate()` automatically whenever its tracked `dataItemCount`
  changes — you don't normally need to invoke it by hand.
- `controller.cancel()` is a subset of `reset()` — it cancels jobs without clearing
  calibration. Use it for "stop loading right now, but keep calibration so the next event
  is treated as a continuation".
- `controller.enabled = false` is a subset of `cancel()` — it pauses dispatch without
  cancelling in-flight jobs. Useful when a modal opens over the list.

## Lifecycle and `cancelOnDispose`

The controller's coroutines run on the `scope` you pass to `paginator.prefetchController(scope = …)`
(or that `bindPaginated` / `rememberPaginated` create internally). The scope is the
authoritative cancellation source — `cancelOnDispose` only adds an early-cancel hook on
top.

| Scope used by the controller  | `cancelOnDispose = true` (default)                                                                                              | `cancelOnDispose = false`                                                                                           |
|-------------------------------|---------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------|
| `viewModelScope`              | Controller is cancelled on Compose `onDispose` / View `ON_DESTROY`. The scope keeps living, but the controller's jobs are gone. | Jobs survive the teardown gap and can complete after the screen returns. **Use this for overlays / bottom-sheets.** |
| `lifecycleScope` / view-bound | Cancelled on teardown.                                                                                                          | The flag has no practical effect — the scope itself is cancelled on `ON_DESTROY`, taking the jobs with it.          |

In short: `cancelOnDispose = false` is only meaningful when the controller's scope outlives
its UI binding (typically a `viewModelScope`). With a lifecycle-bound scope, the scope wins.

## Pitfalls and troubleshooting

If prefetch isn't firing, check in this order:

1. **`totalItemCount` includes non-data items.** Headers, footers, dividers, and append
   loaders inflate the count and push the edge past `prefetchDistance`. The Layer 2+
   bindings exclude them automatically; if you integrate manually (Layer 0 / Layer 1),
   pass data-only counts.
2. **The user hasn't scrolled in the relevant direction yet.** Forward prefetch requires
   `lastVisibleIndex` to have *increased* since the previous `onScroll`. A user who lands
   on the edge and stops will not trigger prefetch — emit a synthetic `onScroll` after
   the first layout pass if you want eager prefetch on appear.
3. **The first event is calibration.** Always. Until a second event arrives, no prefetch
   will fire. Layer 2+ bindings handle this implicitly (a scroll-or-layout pair almost
   always produces ≥ 2 events).
4. **First page is shorter than the viewport.** `OnScrollListener` never fires because
   nothing scrolls. Layer 2+ View bindings also install an `OnLayoutChangeListener`;
   Compose's `snapshotFlow { layoutInfo }` re-emits on layout. If you integrate manually,
   you have to do the equivalent.
5. **`finalPage` reached / `lockGoNextPage` set.** Forward prefetch is gated on
   `endContextPage < finalPage` and `!lockGoNextPage`. Backward needs
   `startContextPage > 1` and `!lockGoPreviousPage`.
6. **`enabled = false`.** Master switch on the controller / `PrefetchOptions`. Master
   switch.
7. **Grid distance is in cells, not rows.** `prefetchDistance = 6` in a 3-column grid
   triggers when 2 rows from the edge. Multiply by your span count when reasoning about
   it.
8. **`scrollSampleMillis` is too aggressive.** Throttling adds up to one window of
   latency. Combined with a small `prefetchDistance`, the user can cross the boundary
   before any `onScroll` reaches the controller.
9. **Reversed layout direction confusion.** The controller works on data indices, not
   visual positions. In a chat (`reverseLayout = true`), forward prefetch fires when the
   user scrolls **upward** toward older messages.
10. **Stale `loadGuard` / `onPrefetchError` lambdas.** Raw lambdas re-create on every
    recomposition and can cause spurious controller reconstructions in low-layer Compose
    setups. Wrap them in `PageLoadGuard` / `CursorLoadGuard` and a stable
    `PrefetchErrorChannel`.

## Testing

The controller is platform-agnostic — call `onScroll` directly with crafted indices, no
UI required:

```kotlin
@Test
fun `forward prefetch fires when within prefetchDistance and scrolling forward`() = runTest {
        val paginator = Paginator<Item>(...)
        val prefetch = paginator.prefetchController(
            scope = backgroundScope,
            prefetchDistance = 5,
        )

        prefetch.onScroll(0, 9, 100)   // calibration — no load
        prefetch.onScroll(1, 10, 100)  // moved forward by 1 — still 89 from end, no load
        prefetch.onScroll(85, 95, 100) // 4 items from the end → triggers goNextPage()

        advanceUntilIdle()
        assertEquals(2, paginator.cache.endContextPage)   // page 2 prefetched
    }
```

Use `runTest` and a `TestScope` for the controller's `scope` so coroutines are deterministic.

## Interaction with other mechanisms

- **Incomplete pages.** If `removeElement()` makes the last loaded page shorter than
  `capacity`, `goNextPage()` re-fetches that page instead of advancing — standard
  paginator behaviour, transparently respected by the prefetch controller.
- **Dirty pages.** CRUD operations with `isDirty = true` mark pages for refresh. The next
  prefetch navigation triggers a fire-and-forget refresh for all dirty pages in the
  context window.
- **Lock flags.** `lockGoNextPage` / `lockGoPreviousPage` block the corresponding prefetch
  direction. Unlocking mid-scroll resumes prefetching on the next `onScroll()` call.
- **`finalPage`.** Forward prefetch stops when `endContextPage >= finalPage`. Changing
  `finalPage` at runtime takes effect on the next scroll check.
- **`resize` / `release`.** After `release()` or a significant `resize()`, call
  `prefetch.reset()` to re-calibrate from the new list state.
