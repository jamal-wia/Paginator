# Prefetch (Auto-Pagination on Scroll)

[← Back to README](../README.md)

> Cursor counterpart: `CursorPaginator.prefetchController(...)` returns a
> `CursorPaginatorPrefetchController` with identical configuration and lifecycle. The only
> difference is the edge check — forward prefetch fires when `endContextCursor?.next != null`,
> backward when `startContextCursor?.prev != null`. See
> [Cursor-Based Pagination → Prefetch](13.%20cursor-pagination.md#prefetch).

`PaginatorPrefetchController` monitors scroll position and automatically calls `goNextPage()` /
`goPreviousPage()` **before** the user reaches the edge of loaded content. This eliminates manual
"load more" buttons and provides a seamless infinite-scroll experience.

The controller is **platform-agnostic** -- it receives visibility information via the `onScroll()`
method, so the same logic works with RecyclerView, LazyColumn, UITableView, or any other list.

## Creating a Controller

```kotlin
val prefetch = paginator.prefetchController(
    scope = viewModelScope,       // coroutine scope for prefetch jobs
    prefetchDistance = 10,         // start loading when 10 items from the edge
    enableBackwardPrefetch = true, // also prefetch when scrolling up (default: false)
)
```

The first `onScroll()` call is **calibration only** -- it records the initial position without
triggering any load, preventing a false positive when the list first appears.

## Android RecyclerView

```kotlin
recyclerView.addOnScrollListener(object : RecyclerView.OnScrollListener() {
    override fun onScrolled(rv: RecyclerView, dx: Int, dy: Int) {
        val lm = rv.layoutManager as LinearLayoutManager
        prefetch.onScroll(
            firstVisibleIndex = lm.findFirstVisibleItemPosition(),
            lastVisibleIndex  = lm.findLastVisibleItemPosition(),
            totalItemCount    = adapter.itemCount,
        )
    }
})
```

## Android RecyclerView (paginator-view)

The optional `paginator-view` artifact removes the `OnScrollListener` plumbing entirely. It
offers three layers of integration — pick whichever one matches the amount of control you need:

```kotlin
dependencies {
  implementation("io.github.jamal-wia:paginator:8.5.0")
  implementation("io.github.jamal-wia:paginator-view:8.5.0")
}
```

| Layer                                            | Tracks `dataItemCount`?             | When to use                                                                               |
|--------------------------------------------------|-------------------------------------|-------------------------------------------------------------------------------------------|
| `paginator.bindPaginated(rv, lifecycleOwner, …)` | **Auto** (from `paginator.uiState`) | Recommended. Zero manual count plumbing.                                                  |
| `paginator.bindPrefetchToRecyclerView(rv, …)`    | Manual lambda                       | You already have your own UI state holder.                                                |
| `controller.bindToRecyclerView(rv, …)`           | Manual lambda                       | You need direct access to the controller (for example, to call `cancel()` from a button). |

Supported `LayoutManager`s in every layer: `LinearLayoutManager`, `GridLayoutManager`,
`StaggeredGridLayoutManager`. Every layer honours the same contract: `totalItemCount` passed to
the controller reflects **only data items** — headers, footers, append loaders, and dividers
are excluded by the binding regardless of how you express them.

Every binding also installs an `OnLayoutChangeListener` in addition to the `OnScrollListener`.
This handles the case where the **first page is shorter than the viewport** and the user can't
scroll — without it, append-pagination would stall on partial first pages. Both listeners are
removed on `ON_DESTROY` automatically.

### Quick start: `bindPaginated`

```kotlin
binding.recyclerView.adapter = ConcatAdapter(headerAdapter, dataAdapter, appendIndicatorAdapter)
binding.recyclerView.layoutManager = LinearLayoutManager(context)

val paged = paginator.bindPaginated(
  recyclerView = binding.recyclerView,
  lifecycleOwner = viewLifecycleOwner,
  headerCount = { headerAdapter.itemCount },
  footerCount = { appendIndicatorAdapter.itemCount },
)
```

What you get for free:

- **`dataItemCount` is auto-tracked** from `paginator.uiState` — only the items count from
  `PaginatorUiState.Content` is observed (transient loading / error states do not trigger
  re-dispatch).
- **The controller is created and lifecycle-scoped** for you (cancelled on `ON_DESTROY` via
  `PrefetchOptions.cancelOnDispose`).
- The returned `PrefetchBinding<PaginatorPrefetchController<T>>` exposes `.controller` (for
  hot-update mutation, `cancel()`, etc.) and the `ScrollBinding` interface (`.recalibrate()`
  after `paginator.refresh()` / `paginator.jump()`, `.unbind()` to detach early).

### One-call: `bindPrefetchToRecyclerView`

If you already track `dataItemCount` in your own UI state holder:

```kotlin
paginator.bindPrefetchToRecyclerView(
  recyclerView = binding.recyclerView,
  lifecycleOwner = viewLifecycleOwner,
  dataItemCount = { uiState.items.size },
  headerCount = 1,
  footerCount = 1,
  options = PrefetchOptions(prefetchDistance = 10, enableBackwardPrefetch = true),
)
```

### Low-level: `prefetchController` + `bindToRecyclerView`

When prefetch lives in a `ViewModel` and the screen only binds it to the `RecyclerView`:

```kotlin
class FeedViewModel : ViewModel() {
  val paginator: Paginator<Item> = ...
  val prefetch = paginator.prefetchController(
    scope = viewModelScope,
    prefetchDistance = 10,
  )
}

class FeedFragment : Fragment() {
  override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    viewModel.prefetch.bindToRecyclerView(
      recyclerView = binding.recyclerView,
      lifecycleOwner = viewLifecycleOwner,
      dataItemCount = { adapter.itemCount },
    )
  }
}
```

### Configuration via `PrefetchOptions`

`PrefetchOptions` is a single immutable bag of runtime settings, shared between
`paginator-compose` and `paginator-view`:

```kotlin
val options = PrefetchOptions(
  prefetchDistance = 10,
  enableBackwardPrefetch = true,
  silentlyLoading = true,
  silentlyResult = false,
  enabled = true,
  cancelOnDispose = true,    // false = controller survives a transient ON_DESTROY
  scrollSampleMillis = 0L,   // > 0 throttles scroll/layout signals via Flow.sample()
)
```

### Reactive errors: `PrefetchErrorChannel`

Compose-style reactive surfacing for prefetch errors, backed by a `StateFlow`:

```kotlin
private val errors = PrefetchErrorChannel()

override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
  paginator.bindPaginated(
    recyclerView = binding.recyclerView,
    lifecycleOwner = viewLifecycleOwner,
    onPrefetchError = errors.onError,
  )

  viewLifecycleOwner.lifecycleScope.launch {
    viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
      errors.errors.collect { e ->
        if (e != null) {
          Snackbar.make(view, e.message ?: "Prefetch failed", LENGTH_SHORT).show()
          errors.consume()
        }
      }
    }
  }
}
```

### Advanced: recalibration and throttling

Every binding returns a `ScrollBinding` (or `PrefetchBinding<C>`, which delegates to
`ScrollBinding`):

- `recalibrate()` — call after a manual `paginator.refresh()` / `paginator.jump()` so the
  controller treats the next event as fresh calibration (the binding does this automatically
  inside `bindPaginated` whenever `dataItemCount` changes).
- `unbind()` — detach scroll / layout listeners early, before `ON_DESTROY`, e.g., when the same
  `RecyclerView` is re-used for a different adapter.
- `scrollSampleMillis > 0` — throttle the scroll-and-layout signal via
  `kotlinx.coroutines.flow.sample`, useful when `loadGuard` or downstream work is non-trivial.

### Stable load guards

`PageLoadGuard` / `CursorLoadGuard` are stable functional interfaces (the same ones used by
`paginator-compose`). Use them to keep guard identity stable across recreations:

```kotlin
private val onlyAfterFirstPage = PageLoadGuard<Item> { page, _ -> page > 1 }

paginator.bindPaginated(
  recyclerView = binding.recyclerView,
  lifecycleOwner = viewLifecycleOwner,
  loadGuard = onlyAfterFirstPage,
)
```

`CursorPaginator` works the same way — every binding has a cursor counterpart with identical
parameters.

## Jetpack Compose LazyColumn

```kotlin
val listState = rememberLazyListState()

LaunchedEffect(listState.firstVisibleItemIndex, listState.layoutInfo) {
    val info = listState.layoutInfo
    prefetch.onScroll(
        firstVisibleIndex = listState.firstVisibleItemIndex,
        lastVisibleIndex  = info.visibleItemsInfo.lastOrNull()?.index ?: 0,
        totalItemCount    = info.totalItemsCount,
    )
}
```

> **Important:** `totalItemCount` must reflect only **data items**. Do not include headers, footers,
> dividers, or loading indicators -- otherwise `prefetchDistance` will trigger inaccurately.

## Jetpack Compose (paginator-compose)

The optional `paginator-compose` artifact removes the `LaunchedEffect` / `snapshotFlow`
plumbing entirely. It offers three layers of integration — pick whichever one matches the
amount of control you need:

```kotlin
dependencies {
  implementation("io.github.jamal-wia:paginator:8.5.0")
  implementation("io.github.jamal-wia:paginator-compose:8.5.0")
}
```

| Layer                                       | Tracks `dataItemCount`?             | Tracks `headerCount` / `footerCount`? | When to use                                                                               |
|---------------------------------------------|-------------------------------------|---------------------------------------|-------------------------------------------------------------------------------------------|
| `rememberPaginated` + `paginated { … }` DSL | **Auto** (from `paginator.uiState`) | **Auto** (from DSL)                   | Recommended. Zero manual numbers.                                                         |
| `PrefetchOnScroll(state, dataItemCount, …)` | Manual                              | Manual                                | You already have your own UI state holder.                                                |
| `rememberPrefetchController` + `BindTo*`    | Manual                              | Manual                                | You need direct access to the controller (for example, to call `cancel()` from a button). |

All three layers honor the same contract: `totalItemCount` passed to the controller reflects
**only data items** — headers, footers, append loaders, and dividers are excluded by the
binding regardless of how you express them.

### Quick start: `rememberPaginated` + DSL

```kotlin
val listState = rememberLazyListState()
val paged = paginator.rememberPaginated(state = listState)

LazyColumn(state = listState) {
  paginated(paged) {
    header { StickyTitle() }
    items(uiState.items, key = { it.id }) { Row(it) }
    appendIndicator { AppendIndicator(uiState.appendState) }
  }
}
```

What you get for free:

- **`dataItemCount`** is derived from `paginator.uiState` via `derivedStateOf` — only
  changes to `Content.items.size` propagate (loading / error state flips do **not** trigger
  a recomposition of the binding).
- **`headerCount` / `footerCount`** are tallied by `header { }` / `appendIndicator { }`
  inside the DSL block. They stay in sync with what you actually rendered, so the classic
  off-by-one bug around the append-loader item is impossible.
- The `paginated` scope behaves like a normal `LazyListScope`; everything else
  (`item`, `items`, `itemsIndexed`, `stickyHeader`) works unchanged.

The same DSL is available for grids and staggered grids:

```kotlin
val pagedGrid = paginator.rememberPaginated(state = gridState)
LazyVerticalGrid(columns = GridCells.Fixed(2), state = gridState) {
  paginated(pagedGrid) {
    header { /* spans the full row by default */ }
    items(uiState.items, key = { it.id }) { Card(it) }
    appendIndicator { Loader() }
  }
}

val pagedStag = paginator.rememberPaginated(state = staggeredState)
LazyVerticalStaggeredGrid(columns = StaggeredGridCells.Fixed(2), state = staggeredState) {
  paginated(pagedStag) {
    header { /* StaggeredGridItemSpan.FullLine by default */ }
    items(uiState.items, key = { it.id }) { Card(it) }
    appendIndicator { Loader() }
  }
}
```

`CursorPaginator` exposes the same `rememberPaginated(state = …)` extensions; the `loadGuard`
parameter takes a `CursorBookmark` instead of a page number.

The holder returned by `rememberPaginated` exposes the underlying controller as
`paged.controller` — useful when you want to invoke `cancel()`, `reset()`, or flip
`enabled = false` from a button.

### One-call: `PrefetchOnScroll`

When you already track item count in your own UI state and just want one composable to wire
prefetch into a lazy state:

```kotlin
val listState = rememberLazyListState()

paginator.PrefetchOnScroll(
  state = listState,
  dataItemCount = uiState.items.size,   // data-only count
  headerCount = 1,                       // optional
  footerCount = 1,                       // optional
    prefetchDistance = 10,
    enableBackwardPrefetch = true,
)

LazyColumn(state = listState) {
    item { StickyTitle() }
    items(uiState.items, key = { it.id }) { Row(it) }
    item { AppendIndicator(uiState.appendState) }
}
```

Overloads cover `LazyListState`, `LazyGridState`, `LazyStaggeredGridState` for both
`Paginator` and `CursorPaginator`. The function returns the constructed
`PrefetchController` if you need a reference.

### Low-level: `rememberPrefetchController` + `BindTo*`

When you want to keep the controller around (for example, to expose a "Pause prefetch"
toggle in the UI) and bind it separately:

```kotlin
val listState = rememberLazyListState()

val prefetch = paginator.rememberPrefetchController(
  prefetchDistance = 10,
  enableBackwardPrefetch = true,
)

prefetch.BindToLazyList(
  listState = listState,
  dataItemCount = uiState.items.size,
  headerCount = 1,
  footerCount = 1,
)
```

Equivalent `BindToLazyGrid` and `BindToLazyStaggeredGrid` overloads exist on both
`PaginatorPrefetchController` and `CursorPaginatorPrefetchController`.

Changing `dataItemCount` / `headerCount` / `footerCount` across recompositions does **not**
restart the underlying flow or re-calibrate the controller — values are read fresh on every
emission. The flow restarts only when the controller identity, the lazy-state instance, or
the optional `restartKey` (see below) changes.

### Configuration via `PrefetchOptions`

`rememberPaginated` accepts an `@Immutable PrefetchOptions` config bag instead of N flat
parameters. Hoist it to share a configuration between screens:

```kotlin
val defaultPrefetch = remember {
  PrefetchOptions(
    prefetchDistance = 12,
    enableBackwardPrefetch = true,
    silentlyResult = true,
    scrollSampleMillis = 16L,   // throttle to one onScroll per ~frame
  )
}

val paged = paginator.rememberPaginated(state = listState, options = defaultPrefetch)
```

`PrefetchOptions` is `@Immutable`, so passing the same instance through recompositions does
not invalidate skippable consumers downstream.

| Field                    | Default                         | Description                                                                                                                                                    |
|--------------------------|---------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `prefetchDistance`       | `DefaultPrefetchDistance` (`6`) | Items from edge that trigger prefetch.                                                                                                                         |
| `enableBackwardPrefetch` | `false`                         | Prefetch toward the start of the list when scrolling up.                                                                                                       |
| `silentlyLoading`        | `true`                          | Suppress `ProgressPage` snapshot during prefetch loading.                                                                                                      |
| `silentlyResult`         | `false`                         | Suppress snapshot when the prefetched page arrives.                                                                                                            |
| `enabled`                | `true`                          | Master switch. `false` makes `onScroll(...)` a no-op.                                                                                                          |
| `cancelOnDispose`        | `true`                          | Cancel in-flight prefetch when the composable leaves composition. Set to `false` to keep prefetch alive across short composition gaps (overlay, bottom-sheet). |
| `scrollSampleMillis`     | `0` (no throttling)             | When `> 0`, throttle the scroll signal via `Flow.sample` so only the latest emission per window reaches the controller.                                        |

### Reactive errors: `PrefetchErrorChannel`

Compose-idiomatic alternative to the imperative `onPrefetchError` callback:

```kotlin
val errors = rememberPrefetchErrorChannel()

paginator.PrefetchOnScroll(
  state = listState,
  dataItemCount = uiState.items.size,
  onPrefetchError = errors.onError,   // stable reference, safe across recompositions
)

errors.current?.let { e ->
  LaunchedEffect(e) {
    snackbar.showSnackbar(e.message ?: "Prefetch failed")
    errors.consume()
  }
}
```

`PrefetchErrorChannel` is `@Stable` and captures `onError` once, so passing it through
recompositions does not re-create the controller.

### Advanced: `restartKey` and `scrollSampleMillis`

**`restartKey`** — bump it (e.g. via a refresh counter) to force the binding to recalibrate
from the current viewport. Useful right after a manual `paginator.refresh()`:

```kotlin
var refreshTick by remember { mutableIntStateOf(0) }

paginator.PrefetchOnScroll(
  state = listState,
  dataItemCount = uiState.items.size,
  restartKey = refreshTick,
)

Button(onClick = {
  paginator.refresh()
  refreshTick++   // restarts the scroll-observation coroutine
}) { Text("Refresh") }
```

`rememberPaginated` accepts the same `restartKey` parameter.

**`scrollSampleMillis`** — when set on `PrefetchOptions` (or passed to `PrefetchOnScroll`
directly), throttles the upstream signal via `kotlinx.coroutines.flow.sample(...)`. Useful
when `loadGuard` does non-trivial work or the list reports very frequent layout updates;
`16L` (one frame) is a reasonable starting value.

### Stable load guards

`rememberPaginated` accepts a `PageLoadGuard<T>` (or `CursorLoadGuard<T>` for cursor-based
paginators) — `fun interface` types whose instances Compose treats as stable, unlike raw
lambdas. This keeps the prefetch controller from being re-created when the surrounding
composable recomposes:

```kotlin
val guard = remember {
  PageLoadGuard<Item> { page, _ -> page < 50 }   // never load past page 50
}

val paged = paginator.rememberPaginated(state = listState, loadGuard = guard)
```

Use the cached `PageLoadGuard.allowAll()` / `CursorLoadGuard.allowAll()` singletons when
no filtering is needed (this is the default).

## Configuration

All properties can be changed at runtime:

| Property                 | Default            | Description                                             |
|--------------------------|--------------------|---------------------------------------------------------|
| `prefetchDistance`       | *(required)*       | Items from edge to trigger load. Must be > 0            |
| `enableBackwardPrefetch` | `false`            | Enable `goPreviousPage()` on scroll up                  |
| `enabled`                | `true`             | Master switch. `false` = `onScroll()` is a no-op        |
| `silentlyLoading`        | `true`             | Suppress `ProgressPage` emission during prefetch        |
| `silentlyResult`         | `false`            | Suppress snapshot emission when prefetched page arrives |
| `loadGuard`              | `{ _, _ -> true }` | Guard callback passed to navigation functions           |
| `onPrefetchError`        | `null`             | Callback for errors (excluding `CancellationException`) |

## Lifecycle Control

```kotlin
// Pause prefetching (e.g. modal is shown), in-flight jobs continue
prefetch.enabled = false

// Cancel in-flight jobs, but new scrolls can still trigger prefetch
prefetch.cancel()

// Full reset: cancel jobs + clear calibration.
// Next onScroll() becomes calibration again (no prefetch on first call).
// Use after jump() or restart() when the list contents change entirely.
prefetch.reset()
```

## Interaction with Other Mechanisms

- **Incomplete pages:** if `removeElement()` makes the last loaded page shorter than `capacity`,
  `goNextPage()` re-fetches that page instead of loading the next one -- this is standard paginator
  behavior and the prefetch controller respects it transparently.
- **Dirty pages:** CRUD operations with `isDirty = true` mark pages for refresh. The next prefetch
  navigation triggers a fire-and-forget refresh for all dirty pages in the context window.
- **Lock flags:** `lockGoNextPage` / `lockGoPreviousPage` block the corresponding prefetch
  direction.
  Unlocking mid-scroll resumes prefetching on the next `onScroll()` call.
- **`finalPage`:** forward prefetch stops when `endContextPage >= finalPage`. Changing `finalPage`
  at runtime immediately affects the next scroll check.
- **`resize` / `release`:** after `release()` or significant `resize()`, call `prefetch.reset()` to
  re-calibrate from the new list state.
